Section 02 — Project 1 · Full Notes (Part 1 + Part 2)

Build a Character AI Chatbot

Complete project walkthrough: environment setup, your first OpenAI API call, decoding the response object, tokens, giving the model a persona, and the module recap.

Reading lens — Two ideas carry the whole project: (1) the Chat Completions API is stateless — your client owns and resends the full conversation every call — and (2) the system message is runtime configuration, not conversation — a persona/behavior payload selected from a lookup table and injected alongside the user's message. Everything else (models, tokens, response metadata) is detail sitting on top of those two facts.

API key handling:

  • Load via python-dotenv: load_dotenv() → os.environ.get("OPENAI_API_KEY")

Why it matters — Same discipline as a vault/secrets-manager pattern: never commit, never hardcode, load at runtime.

Architect's note — This endpoint is stateless. There's no server-side session — your client resends the entire messages history on every call. This is the design root of why chat-app token cost grows with conversation length.

TASK 06

Response Object — full anatomy

6
Beyond .choices[0].message.content
id · model · finish_reason · usage · fingerprint
›
id
Unique response identifier — like a request/correlation ID for tracing.
model
Confirms which model actually served the response.
role: assistant
Marks the reply as AI-generated, mirroring your role: user.
finish_reason
"stop" = natural completion; other values signal truncation (e.g. max tokens).
usage.prompt_tokens
Tokens in your input — the part you control.
usage.completion_tokens
Tokens the model generated in reply.
usage.total_tokens
Sum of both — your literal billing line item.
refusal / system_fingerprint
Populated if the model declines on policy grounds; fingerprint is internal OpenAI tracking metadata.
Cost lens — Since full history is resent each call, prompt_tokens grows every turn in a multi-turn chat — the concrete mechanism behind runaway chat costs in production.
TASK 07–08

Tokens, the Tokenizer & the politeness cost

7·8
Tokenization + practice solution
13 vs. 17 tokens
›

A token is a text chunk (word/sub-word); a tokenizer converts text → tokens → numeric token IDs, the actual input to the model's matrix of weights.

IloveDoctorRyan'sbootcamponLLM

Rule of thumb: 1 token ≈ 4 characters ≈ ¾ of a word → 100 tokens ≈ 75 words. Try it: OpenAI Tokenizer ↗

Practice result: "Explain the difference between supervised and unsupervised learning" = 13 tokens. Adding "Please" / "Thank you." = 17 tokens — same meaning, +4 tokens purely from politeness. At API scale this is the real mechanism behind Sam Altman's "please and thank you cost millions" comment — metered GPU compute per token, multiplied across millions of calls.

Architect's note — Token count ≈ payload size for latency and cost — same discipline as minimizing request/response body size in a high-throughput API.

The Chat Completions API is stateless. What does that mean for building a multi-turn conversation??
Your client must resend the entire message history — every prior user/assistant turn — on every call. The server holds no session.
Where does the API key live, and how is it loaded at runtime??
In a local .env file (never hardcoded/committed), loaded via python-dotenv's load_dotenv() and read with os.environ.get("OPENAI_API_KEY").
What are the three token counts in usage, and which one is your bill??
prompt_tokens (input), completion_tokens (output), total_tokens (sum) — total_tokens is what you're billed per call.
How is a persona stored and selected, and what actually changes in the API call to use one??
A Python dict maps persona name → instruction string; the chosen value is injected as a second message with role: "system". Model, method call, and response parsing stay identical.
Plain vs. polite prompt token counts — and why does it matter at scale??
13 tokens plain vs. 17 with "please"/"thank you" — same meaning, more tokens. At millions of API calls, that's real, multiplied compute cost.
