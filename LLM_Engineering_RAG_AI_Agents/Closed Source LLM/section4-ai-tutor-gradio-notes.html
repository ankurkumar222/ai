Section 04 — Project 3

AI Tutor with Gradio

First real UI layer on top of the API fundamentals: wrap a Python function in a shareable web interface, add real-time streaming, and layer in a slider that dynamically adjusts the AI's explanation complexity.

Reading lens — Gradio is a declarative UI framework — you describe a function's input/output shape (textbox, slider) and it generates the interface, closer to Swagger/OpenAPI auto-generating a UI from a schema than to hand-rolling frontend code. Streaming is the same chunked-response / SSE pattern you already know from any server pushing partial data before a full response completes — here expressed via Python's yield generator keyword.

What is Gradio, and setup
pip install gradio + OpenAI client config (same as before)
›

Gradio ↗ — an open-source library (backed by Hugging Face; used at Amazon, Facebook, Cisco, etc.) that turns a Python function into a web interface in ~5 lines. Three key properties:

  • Easy setup — pip install gradio, then import gradio as gr.
  • Instant sharing — runs embedded in a notebook or as a separate tab with a shareable link.
  • Permanent hosting — deployable to Hugging Face Spaces ↗ for a persistent public URL.
import gradio as gr def greet(name): return f"Hello {name}!" demo = gr.Interface(fn=greet, inputs="text", outputs="text") demo.launch()

OpenAI client setup is unchanged from prior projects — .env key, load_dotenv(), configure the client, sanity-check the key prefix.

Note — Gradio's example gallery ↗ includes maps, streaming chatbots, and image-generation demos — same Interface(fn, inputs, outputs) pattern scales from "hello world" to fairly complex apps.
AI Philosophy — Gradio is an early instance of a shift that's accelerating across the whole stack: function signatures becoming the API surface for UI generation. The same pattern recurs at every layer — prompts becoming the API surface for reasoning, JSON schemas becoming the API surface for agent tool-use. The unit of composition keeps moving up the abstraction ladder, trading raw expressiveness for speed of iteration. The skill to develop is knowing which layer is the right one for each job, and resisting the pull to drop a level when the current one is sufficient.
3
Basic AI Tutor Function (no UI yet)
get_ai_tutor_response() — system persona + temperature
›

Build the core logic first, decoupled from any interface — a reusable, testable function before wiring UI on top.

def get_ai_tutor_response(question: str) -> str: """Sends the question to OpenAI, asking it to respond as an AI tutor.""" system_prompt = "You are a helpful and patient AI tutor. Explain things clearly and concisely." try: response = client.chat.completions.create( model="gpt-4o-mini", messages=[ {"role": "system", "content": system_prompt}, {"role": "user", "content": question}, ], temperature=0.7, ) return response.choices[0].message.content except Exception: return "Sorry, I encountered an error trying to get an answer."

temperature — a knob between 0–2 controlling output randomness/creativity. Higher = more varied/creative; lower = more deterministic/focused. 0.7 is a moderate default here.

Architect angle — try/except around the API call is basic fault tolerance for an external dependency — same instinct as wrapping any third-party network call, since the API can fail, time out, or rate-limit independently of your code.
AI Philosophy — Building the core logic function before adding any UI is more than a code-organisation preference — it's a testability discipline. A function decoupled from its interface can be unit-tested, swapped between providers, benchmarked on a prompt eval set, and handed to a teammate without requiring them to run a notebook. In AI engineering, where iteration on the logic layer is constant, that seam between "what the model does" and "how the user reaches it" pays for itself faster than almost any other structural choice.

Exercise: swap the test question (e.g. gravity, photosynthesis), then edit only the system_prompt string to change personality (e.g. "impatient and unhelpful," "angry and sad"). Function signature and call logic stay identical — persona is purely a config-string swap.

6
Wrapping the Function in a Gradio Interface
gr.Interface(fn, inputs, outputs, title, description)
›

Gradio needs exactly four things to turn a function into a UI:

fnThe Python function to call — here, get_ai_tutor_response.
inputsgr.Textbox(lines=2, placeholder="Ask the AI tutor anything", label="Your question")
outputsgr.Textbox(label="AI Tutor's Answer")
title/descOptional display text above the interface.
tutor_interface = gr.Interface( fn=get_ai_tutor_response, inputs=gr.Textbox(lines=2, placeholder="Ask the AI tutor anything", label="Your question"), outputs=gr.Textbox(label="AI Tutor's Answer"), title="Simple AI Tutor", description="Enter your question and the AI tutor will explain it, powered by OpenAI.", ) tutor_interface.launch()

Launch produces a real, shareable link — no more "open Jupyter, hit Shift+Enter" for a non-technical user to try the app.

Architect angle — This is UI generated purely from a function signature's input/output shape, not hand-built markup — think of it as the lowest-effort way to put an internal tool or POC in front of a stakeholder, roughly the AI-engineering equivalent of an auto-generated admin panel. The shareable link also matters: a non-technical stakeholder can interact with your prototype without installing anything, which compresses the feedback loop from days to hours.

9
Enabling Streaming
stream=True + yield → the function becomes a generator
›

Three changes turn the plain function into a streaming one:

1 · stream=True
Tells the OpenAI API to return the response as a series of chunks instead of one complete payload.
2 · yield instead of return
Turns the function into a Python generator — each yield hands a partial result back to the caller without ending the function.
3 · Gradio handles the rest
Point gr.Interface's fn at the generator function — Gradio automatically re-renders the output as new chunks arrive.
def stream_ai_tutor_response(question: str): system_prompt = "You are a helpful and patient AI tutor. Explain concepts clearly and concisely." stream = client.chat.completions.create( model="gpt-4o-mini", messages=[ {"role": "system", "content": system_prompt}, {"role": "user", "content": question}, ], temperature=0.7, stream=True, ) full_response = "" for chunk in stream: if chunk.choices[0].delta.content: full_response += chunk.choices[0].delta.content yield full_response # hand the growing string back each time

Note full_response accumulates and is re-yielded in full each time (not just the new delta) — Gradio expects each yield to be the complete current output, not an incremental diff.

Architect angle — This is the same mechanic as Server-Sent Events or HTTP chunked transfer encoding — the server (OpenAI) pushes partial data as it's produced instead of buffering the full response. yield is Python's native tool for exposing that as an iterable stream to the caller.
AI Philosophy — Perceived latency and actual latency are separate engineering problems, and they often deserve separate solutions. Streaming is effectively free UX improvement — same compute, same cost, dramatically better perceived responsiveness. But it also changes what "done" means in your state model: instead of a single completion event, you now have a live stream that must be displayed, possibly interrupted, and error-handled mid-flight. That complexity is worth it at any non-trivial response length. Get comfortable with the generator pattern early — agentic pipelines, tool-call loops, and multi-step reasoning chains all produce output the same way.
10·11
Practice — streaming vs. non-streaming UX
Same long prompt, both interfaces, compare perceived latency
›

Send an identical multi-paragraph prompt (e.g. "explain in detail how EVs work") to both the streaming and non-streaming interfaces. Non-streaming takes roughly ~6 seconds of dead wait before any text appears; streaming shows text within roughly a second and keeps building — same total generation time, dramatically different perceived latency.

Takeaway — Total response time is identical either way; only perceived responsiveness changes. This is the same principle behind progress bars and skeleton screens in traditional UI — streaming is effectively free UX improvement for any non-trivial LLM response length.

The Explanation-Level Slider
Dict-mapped levels 1–5 → dynamic system prompt
›

Three-part pattern, same lookup-table shape as the Project 1 personas:

explanation_levels = { 1: "as if I'm 5 years old", 2: "as if I'm 10 years old", 3: "clearly and concisely", 4: "in more technical depth", 5: "at an expert level", }

The streaming function now takes two inputs — question and explanation_level (an int) — and interpolates the mapped description straight into the system prompt:

def stream_ai_tutor_response_with_level(question: str, explanation_level: int): level_description = explanation_levels.get(explanation_level, "clearly and concisely") system_prompt = f"You are a helpful AI tutor. Explain the following concept {level_description}." # ...same client.chat.completions.create(..., stream=True) + yield loop as Task 9

Gradio side adds a gr.Slider alongside the existing textbox:

gr.Interface( fn=stream_ai_tutor_response_with_level, inputs=[ gr.Textbox(lines=3, placeholder="Ask the AI tutor anything", label="Your question"), gr.Slider(minimum=1, maximum=5, step=1, value=3, label="Explanation level"), ], outputs=gr.Markdown(label="AI Tutor's Explanation"), title="Advanced AI Tutor", allow_flagging="never", ).launch()

.get(explanation_level, "clearly and concisely") provides a safe fallback if an unmapped value ever arrives — the same defensive-default pattern as Task 3's try/except.

Architect angle — inputs is now a list of components matching the function's positional arguments in order — Gradio wires UI components to function parameters purely by position, the same convention as unpacking args into a call.
AI Philosophy — A dict mapping levels to prompt fragments is a primitive behavior manifest — configuration that describes what the AI should do, separated from the code that calls it. This separation scales. LLM applications increasingly split into two distinct layers: the behavior layer (prompts, personas, level descriptions), which domain experts and product managers can own and iterate on, and the infrastructure layer (API calls, streaming logic, routing), which engineers own. Teams that establish this boundary early can change AI behavior without touching code — and hand the behavior layer to the people who actually understand the domain, regardless of whether they can write Python.
13·14
Practice — extend to level 6
"PhD-level Einstein mad scientist" + relaunch
›

Two coordinated changes required — both the data and the UI config must extend together:

1 · Extend the dict
explanation_levels[6] = "as an Einstein-level PhD mad scientist"
2 · Extend the slider range
gr.Slider(minimum=1, maximum=6, ...) — the dict key alone isn't enough; the slider's own maximum must be raised or level 6 is simply unreachable in the UI.

Level 6 produces noticeably denser, more ornate language ("let's embark on a journey through the electric ether...") versus level 1's toy-box simplicity — the prompt-level lever scales smoothly across the whole range.

Common gotcha — Forgetting to bump the slider's maximum alongside a new dict entry is an easy mismatch bug — the config (UI bounds) and the data (lookup table) have to be kept in sync manually since nothing here derives one from the other automatically.

Active Recall Check

Click a card to reveal the answer
What four things does gr.Interface minimally need to turn a function into a web UI??
fn (the Python function), inputs (component(s) matching its parameters), outputs (component(s) matching its return value), and optionally title/description.
What two code changes convert a normal response function into a streaming one??
Set stream=True in the API call, and replace return with yield (turning the function into a generator that hands back the growing response incrementally).
Does streaming reduce total response time??
No — total generation time is roughly the same. What improves is perceived latency: text appears within ~1 second instead of a multi-second blank wait.
How does the explanation-level slider actually change the AI's behavior under the hood??
The selected integer is looked up in a dict mapping level → description phrase, which is string-interpolated into the system prompt sent to the API — no model or code logic changes, only the prompt text.
When adding a new level 6 to the dictionary, what second change is also required??
The Gradio slider's maximum parameter must also be raised (e.g. 5 → 6) — otherwise the new dict entry is unreachable from the UI.
How does inputs=[...] map to a function's parameters when there are multiple??
By position — the first component in the list feeds the function's first parameter, the second component feeds the second parameter, and so on.
