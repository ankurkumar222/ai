Section 03 — Project 2

Build an AI Calorie Tracker

Vision API + prompt engineering fundamentals: feed the model a food photo, get back structured nutrition data. Covers image handling, the 4-part prompt framework, zero/few-shot & chain-of-thought, base64 encoding, and prompt-enforced structured output.

Reading lens — Two threads run through this whole project: (1) vision input is just another payload format — an image becomes a base64 string embedded in the same messages JSON you already know from Project 1; (2) prompt engineering is API contract design done in natural language — context/instruction/input/output-indicator is you hand-writing a request/response schema before you have Pydantic to enforce it for you.

Loading & Inspecting an Image (Pillow)
Format, size, mode — before any API call happens
›

Pillow (PIL) is Python's image-handling library. Load and inspect an image locally before sending anything to an API — a basic sanity check on your input.

from PIL import Image image_file_name = "images/food_image.jpg" image = Image.open(image_file_name) print("Format:", image.format) # JPEG print("Size:", image.size) # (600, 400) print("Mode:", image.mode) # RGB — color image image_to_analyze = image # keep a reference for later encoding

Mode = RGB means a full-color image (red/green/blue channels) — as opposed to grayscale.

4
Prompt Engineering Fundamentals
The 4-part prompt framework + 3 core techniques
›

Prompt engineering = designing/optimizing text input to a generative model for consistent output. Specificity compounds: "recommend a book" → "recommend a fiction book" → "recommend a historical fiction book set in WWII" — each refinement narrows the output space.

The 4-part structure (sentiment-classification example):

Context
"Assume you are an expert financial analyst" — sets the persona/scene.
Instruction
"Classify the following text as positive, neutral, or negative" — the exact task.
Input
"Apple stock increased by 2% in today's trading session" — the actual data.
Output indicator
"Sentiment:" — tells the model the exact shape of the expected answer.

Three prompting techniques:

Zero-shot

Ask the model to perform a task with no examples in the prompt — relies purely on the model's pretrained knowledge.

"Categorize this headline as positive/negative/neutral: ..."

Few-shot

Provide example input→output pairs inside the prompt itself (not fine-tuning) to steer the model toward a pattern.

"Apple -2% → negative. Tesla +5% → positive. OpenAI +100M users → ?"

Chain-of-thought

Break a complex task into explicit intermediate reasoning steps in the prompt — dramatically improves accuracy on multi-step problems (esp. older/smaller models).

"Sarah started with 10... gave away 4, leaving 6... bought 5 more, 6+5=11."
Trade-off — Few-shot and chain-of-thought both increase token count (= cost + latency) in exchange for reliability. This is the same lever you'll weigh later when choosing between prompting, RAG, and fine-tuning for a given accuracy bar.

6
Sending Images to the API — base64 encoding
Two helper functions: encode_image_to_base64, query_openai_vision
›

OpenAI's vision endpoint accepts images as a URL or a base64-encoded string. Since a JSON payload is text-only, binary image bytes must be serialized to text first.

import base64 def encode_image_to_base64(image_input): if isinstance(image_input, str): # file path with open(image_input, "rb") as f: return base64.b64encode(f.read()).decode() else: # PIL Image object buffer = io.BytesIO() image_input.save(buffer, format="JPEG") return base64.b64encode(buffer.getvalue()).decode()

Second function, query_openai_vision(client, image, prompt, model="gpt-4o", max_tokens=100), wraps the full call: encodes the image, builds the message payload, calls the API, returns the text response. Defaults (model="gpt-4o", max_tokens=100) mean callers only need to pass what differs per-call.

messages = [{ "role": "user", "content": [ {"type": "text", "text": prompt}, {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{base64_image}"}} ] }]
Architect's note — Notice content is now a list of typed parts (text, image_url) instead of a plain string — a multimodal message is a heterogeneous payload, same shape as a multipart form or a GraphQL union type. The role/messages envelope from Project 1 didn't change; only what can go inside content expanded.
Function design note — Default parameter values (model="gpt-4o") are exactly what you'd expect from any well-designed config object — sane defaults, override only what's different for a given call.

9
Structured Nutrition Output — the core deliverable
Prompt-enforced JSON, pre-Pydantic
›

The final prompt specifies an exact output schema in natural language, field by field, with types called out explicitly and a worked example (one-shot) to lock the format:

Output — respond only with a single JSON object containing: food_name (string) serving_description (string, e.g. "1 cup", "100g", "1 slice") calories (float) fat_grams (float) protein_grams (float) confidence_level (string: "high" / "medium" / "low") Example: { "food_name": "banana", "serving_description": "1 medium", "calories": 105, "fat_grams": 0.4, "protein_grams": 1.3, "confidence_level": "high" }

Result on the salmon image: calories: ~350, fat_grams: ~20, protein_grams: ~35, confidence_level: "medium" — a full structured estimate from a single API call.

Big picture — This is "structured output via prompting" — the model is instructed to self-format as JSON, with no schema enforcement beyond the instruction text and a one-shot example. It works, but it's fragile (the model could still return prose, markdown fences, or malformed JSON). This is explicitly framed as the naive precursor to Pydantic-based output validation covered later in the bootcamp — the difference between "asking nicely for a shape" and "enforcing a schema."
10
Practice — extend fields, stress-test robustness
sugar_grams, fiber_grams · pizza vs. mixed Greek salad
›

Add sugar_grams and fiber_grams (both float) to the schema and one-shot example, then test on harder images:

Pizza slice (single, common food)
1 slice ≈ 107g, ~298 calories, confidence: high. A single, extremely common item — the model has seen countless reference images.
Greek salad (mixed dish)
1 bowl ≈ 200 calories, protein ~5g, sugar ~5g, fiber ~4g, confidence: medium. Multiple co-mingled ingredients reduce the model's own reported confidence.
Key observation — The model's self-reported confidence_level tracked visual complexity — single common foods scored high, multi-ingredient mixed dishes scored medium. This is a useful, if soft, signal for downstream logic (e.g. flag low-confidence estimates for user review) — and it also means every API call can return slightly different numbers, since nothing here is deterministic.

Active Recall Check

Click a card to reveal the answer
What are the four components of a well-structured prompt??
Context (persona/scene) → Instruction (the task) → Input (the data) → Output indicator (the expected answer shape).
Distinguish zero-shot, few-shot, and chain-of-thought prompting.?
Zero-shot: no examples, rely on pretrained knowledge. Few-shot: example input→output pairs embedded in the prompt (not fine-tuning). Chain-of-thought: explicit step-by-step reasoning written into the prompt to improve multi-step accuracy.
Why does an image need to be base64-encoded before an API call??
The API payload is JSON (text-only); base64 encoding serializes binary image bytes into a text string that can be embedded in that JSON structure.
How does a multimodal message's content field differ from the plain-text version in Project 1??
content becomes a list of typed parts — e.g. {"type": "text", ...} and {"type": "image_url", ...} — instead of a single string, so text and image can travel in the same message.
How was structured (JSON-shaped) output achieved in this project, and what's the limitation of that approach??
By explicitly specifying the field names/types and a one-shot example in the prompt itself, and instructing the model to respond with only a JSON object. Limitation: there's no real schema enforcement — the model could still return malformed or non-JSON text. Pydantic-based validation is the more robust successor covered later.
What pattern did the confidence_level field reveal when testing pizza vs. a mixed Greek salad??
Simple, common, single-item foods (pizza slice) got high confidence; complex, multi-ingredient mixed dishes (Greek salad) got medium confidence — the model's self-reported certainty tracked visual/compositional complexity.
