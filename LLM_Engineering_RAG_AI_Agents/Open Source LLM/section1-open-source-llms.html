Part one

The open-weight decision

Before any code: what actually changes when the model lives on your disk instead of someone else's server. This is a procurement and architecture decision disguised as a technical one.

01

What changes when you own the weights

In Section 5 you called GPT-4o, Gemini and Claude. You sent text over HTTPS and got text back. You never touched the model. Here, you download the model itself — billions of numbers — and run it on hardware you're responsible for.

The five real reasons to self-host

01 · Data residency

Nothing leaves the perimeter

The decisive one in banking, healthcare and defence. If customer PII, KYC documents or trade data cannot legally cross into a third-party API, capability comparisons are irrelevant — self-hosting is the only compliant path.

02 · Unit economics at volume

Fixed cost beats per-token

API pricing is linear in usage. A GPU is a fixed cost. Below a certain volume the API always wins; above a crossover point, self-hosting wins. Know where your crossover is before arguing about it.

03 · Customisation

Fine-tuning is only possible here

You can continue training on your own domain data — your product catalogue, your regulatory language, your ticket history. No closed API gives you deep weight-level adaptation.

04 · Version stability

The model doesn't change under you

A downloaded checkpoint is frozen forever. Hosted models get silently updated and deprecated, which can break carefully tuned prompts overnight. For anything you must audit or reproduce, this matters enormously.

05 · Offline & edge

Works with no internet

Air-gapped environments, factory floors, on-device inference, disaster-recovery sites. Not possible with an API by definition.

Bonus · Transparency

You can inspect it

Architecture, layer counts, attention shapes are all visible. Useful for research, for security review, and for explaining behaviour to a regulator who asks "how does it work?"

The honest counter-argument

Most teams that self-host underestimate the total cost. You are now paying for: GPU capacity that sits idle between requests, an inference server, model-update evaluation, monitoring, a security review of downloaded artefacts, and engineers who understand CUDA errors at 2am. A frontier API is often cheaper and better until you hit either a compliance wall or serious volume. "We should self-host to save money" is the most common wrong reason.

Name the one reason to self-host that no amount of API improvement can ever overturn.

Data residency / legal constraint. Cost, quality and latency arguments all shift as vendors improve and prices fall. But if the law or your contract says the data cannot leave your network, no API — however cheap or brilliant — becomes eligible. Every other reason is a moving target; this one is a hard boundary.

Your CFO asks: "at what point does self-hosting actually become cheaper?" Sketch how you'd answer.

Set up a break-even, not an opinion. API cost = (tokens per request × requests per month) × blended price per token. Self-host cost = GPU instance hours per month + engineering time amortised + storage/egress. Then solve for the request volume where they cross.

Two things people forget. Utilisation: a GPU you rent 24×7 but use 6% of the time makes your effective per-token cost enormous — the API's elasticity is a real feature you're giving up. Quality per rupee: if the open model needs three retries to match one frontier-model answer, the token count in your comparison is wrong.

The honest answer usually is: "cheaper above roughly X million tokens a month at Y% GPU utilisation, and we're currently at 5% of X." Which reframes the conversation to the reason that actually applies — compliance or customisation.

Frozen weights are listed above as an advantage. When are they a serious liability?

The moment safety, accuracy or capability improves elsewhere and you don't get it. A frozen checkpoint keeps every bias, jailbreak and factual error it shipped with, forever. Hosted models get patched; yours doesn't unless you re-evaluate, re-download and re-test.

So "version stability" isn't free — it converts an implicit vendor responsibility into an explicit internal process. If you self-host and never revisit the model, you haven't chosen stability; you've chosen decay. The mature version is a deliberate upgrade cadence: pinned in production, evaluated quarterly against your own test set.

You load a model and it rambles instead of answering your question. What's the first thing to check?

Whether you grabbed the base model rather than the instruct variant. Base models are trained purely to continue text — asked "What is the capital of France?" they may cheerfully generate five more exam questions. Instruction-tuned versions (-instruct, -it, -chat) have been further trained to treat your input as a request.

Second thing to check: whether you applied the model's chat template. Right model, wrong formatting produces the same symptom.

Why is "1.6 million models available" a problem as much as a feature?

Because selection cost becomes the bottleneck, and most of those checkpoints are unmaintained forks, experiments and abandoned fine-tunes. An open registry with no gatekeeping gives you both extraordinary choice and no quality signal.

The practical response is to narrow hard before you browse: filter by task, then by organisation you'd trust to still exist next year (Meta, Google, Microsoft, Alibaba, Mistral, DeepSeek), then by licence, then by size that fits your hardware. That takes 1.6 million down to about a dozen. Treat model selection like dependency selection — you would not install a random npm package with four stars into a payments system, and a fine-tune is a far larger trust surface than a package.

What is the supply-chain security question nobody in the tutorial asks?

You are downloading executable-adjacent artefacts from a public registry and loading them into a process with GPU and filesystem access. The questions a security review will ask:

Format: is it safetensors or legacy pickle? Pickle files can execute arbitrary code on load. Prefer safetensors, always.
Remote code: does it require trust_remote_code=True? That flag means "run Python from this repo on my machine" — see section 09.
Provenance: is the repo the official org, or a lookalike? Typosquatting exists here too.
Pinning: are you pulling main, or a specific commit revision? Unpinned means the artefact can change under you.
Mirroring: in a regulated shop, models should be pulled once, scanned, and served from an internal registry — never fetched from the internet at container start.

The parameter-count intuition

B = billions of weights. Qwen3-32B has 32 billion numbers in it. This single figure predicts your GPU bill, your latency and roughly your capability — which is why it's in the name. A useful ladder: 0.5–3B runs on a laptop or a free Colab GPU and handles extraction, classification and simple Q&A. 7–14B is the sweet spot for most self-hosted production work. 30B+ needs serious hardware and starts approaching frontier quality on narrow tasks.

Two candidates: Phi-3-mini-4k-instruct and Phi-3-mini-128k-instruct. Same family, same size. When does the 4k version still win?

When your inputs are short. Longer context isn't free — it costs VRAM for the KV cache (section 06), it costs latency, and long-context variants are sometimes slightly weaker on short-input tasks because capacity was spent on length. If you're classifying support tickets of 200 tokens, buying a 128k window is paying for a warehouse to store a envelope.

The 128k version wins the moment a single document genuinely can't be split without losing meaning.

You see -GGUF and -AWQ variants of a model you like. What does their existence tell you about the model's real-world usage?

That people are actually deploying it on constrained hardware — those are community quantization formats produced because there's demand. Their presence is a decent proxy for "this model has an ecosystem": tooling support, inference-engine compatibility, people who've hit and documented the bugs.

It also tells you something about your own next step. If a well-maintained quantized build already exists, you may not need to quantize at load time yourself — you can pull a pre-quantized artefact that someone has already validated, which is faster to load and often better tuned than a naive on-the-fly conversion.

Part two

Hardware is the real constraint

With APIs, capability was the constraint. Here, memory is. This part is the difference between "the model I want" and "the model I can actually run" — and it's where most self-hosting projects die.

04

Why a GPU, and which one

Running an LLM is, mechanically, an enormous pile of matrix multiplications. Every generated token requires passing a vector through billions of weights. CPUs execute a few operations at a time, very fast. GPUs execute thousands simultaneously, somewhat slower each. For this workload, width beats speed by orders of magnitude.

   [====>]                             [=>][=>][=>][=>][=>][=>][=>][=>]

VRAM matters more than speed

Notice the trap: people compare GPUs on throughput, but the thing that stops you is capacity. A slow GPU with enough VRAM runs your model; a fast GPU without enough VRAM throws CUDA out of memory and runs nothing. When choosing hardware for inference, check that the model fits first, then optimise for speed.

In Colab: Runtime → Change runtime type → Hardware accelerator. Then verify from code rather than trusting the menu:

Always verify the GPU before loading anything
import torch

if torch.cuda.is_available():
    name = torch.cuda.get_device_name(0)
    total = torch.cuda.get_device_properties(0).total_memory / 1024**3
    print(f"GPU: {name} — {total:.1f} GB VRAM")
else:
    print("No GPU. Loading a 3B model here will take minutes per token.")
Your notebook idles for twenty minutes, then every cell fails with "torch is not defined". What happened?

Colab reclaimed your runtime. The VM was recycled, so every import, variable, downloaded model and logged-in session is gone. You have to re-run from the top — install, import, log in, reload the model (which re-downloads gigabytes).

The engineering lesson beyond the annoyance: your notebook state is not durable infrastructure. Anything that takes five minutes to rebuild and lives only in RAM is a demo, not a system. Real deployments load the model once into a long-lived process precisely to avoid this cost.

Why would a production team accept a slower GPU than the one their benchmark says is fastest?

Because inference cost is dominated by utilisation, not peak speed. A cheaper GPU that stays 70% busy beats an A100 at 8% busy on cost-per-query, every time. Add availability (top-tier GPUs are frequently unobtainable in your region), quota limits, and the fact that a bigger card only helps if your model and batch actually fill it.

The real question is never "which GPU is fastest?" but "what is my cost per thousand queries at realistic traffic, including idle time?" That reframing usually points at smaller cards, aggressive batching, and autoscaling to zero — or, honestly, back at the API.

05

Precision and quantization

Every weight is a number, and you choose how many bits to store it in. That choice is the single biggest lever you have over whether a model fits on your hardware.

Sketch 5 · The precision ladder
   FORMAT   BITS  BYTES/WEIGHT   7B MODEL   WHAT YOU GIVE UP
   ------------------------------------------------------------------
   fp32      32      4.0          ~26 GB    nothing (training default)
   fp16/bf16 16      2.0          ~13 GB    almost nothing for inference
   int8       8      1.0          ~6.5 GB   very little
   int4       4      0.5          ~3.5 GB   a little; usually acceptable
                                            ^^^^^^^^^^^^^^^^^^^^^^^^^
   THE FORMULA YOU SHOULD MEMORISE
   -------------------------------------------------------------
       VRAM for weights (GB)  ~=  params(B)  x  bytes-per-weight

       7B  x 2 bytes (fp16) = 14 GB   -> will NOT fit a 15GB T4 with
                                         room for anything else
       7B  x 0.5 (int4)     = 3.5 GB  -> fits comfortably

How you actually ask for it

4-bit quantization via bitsandbytes
from transformers import AutoModelForCausalLM, BitsAndBytesConfig
import torch

bnb = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",              # NF4 > plain int4 for LLM weights
    bnb_4bit_compute_dtype=torch.float16,   # maths still runs in fp16
    bnb_4bit_use_double_quant=True,        # quantize the quantization constants too
)

model = AutoModelForCausalLM.from_pretrained(
    model_id,
    quantization_config=bnb,
    device_map="auto",
)
The subtlety worth understanding

Storage precision and compute precision are different things. Weights sit in memory at 4 bits, but they're dequantized to fp16 to do the actual matrix multiply. That's why bnb_4bit_compute_dtype exists. It also explains why 4-bit saves enormous memory but doesn't make inference proportionally faster — you've reduced what you store, not the arithmetic you perform. Quantization is a memory optimisation that buys you access, not a speed optimisation.

You quantize to 4-bit and quality drops noticeably on your task. What are your options, in order of cost?

1. Step up to 8-bit. Roughly doubles memory but usually recovers most of the loss. Cheapest fix if it fits.

2. Try a better quantization scheme. NF4 with double quantization, or a community AWQ/GPTQ build that was calibrated properly, often beats a naive conversion at the same bit width.

3. Drop to a smaller model at higher precision. A 3B at fp16 frequently beats a 7B crushed to 4-bit — same memory, better behaviour. This trade is underused.

4. Rent a bigger GPU. The honest last resort, and sometimes the right one.

Quantization is lossy. Why doesn't it destroy the model the way lossy compression destroys a spreadsheet?

Because neural networks are massively redundant and inherently noise-tolerant. No single weight carries decisive information; the behaviour emerges from patterns across billions of them. Small rounding errors distributed across all weights largely cancel rather than compound — the network was trained with noise (dropout, stochastic gradients) and is robust to more of it.

A spreadsheet is the opposite: every cell is load-bearing, and one wrong digit is one wrong answer. Redundancy is what makes compression survivable, and it's why the same trick doesn't transfer to systems where each value matters individually.

Follow-on worth sitting with: this is also why quantization damage isn't uniform. It tends to show up first in the rarest, most precise behaviours — long arithmetic, exact recall of obscure facts, low-resource languages — while fluent conversation stays intact. So a quantized model can feel fine in casual testing and fail on exactly the edge cases you care about.

When is running a quantized model actively the wrong call, even though it fits?

When the failure mode is silent and the stakes are high. Quantization degrades precision-sensitive behaviour first — numerical reasoning, exact figure extraction, rare-token handling. In a financial-document pipeline that pulls numbers out of filings, a 4-bit model that transposes a digit produces a plausible, fluent, wrong answer that no error handler catches.

The test isn't "does it still sound good." It's "does it still get the numbers right, measured on my own labelled set." If you haven't measured that specifically, you don't know what quantization cost you — you only know it loads.

06

The memory budget nobody warns you about

Weights are the number everyone quotes. They're not the only thing in VRAM, and the other components are the ones that make your app fail in production but work in testing.

Sketch 6 · What's actually sitting in your GPU
   +--------------------------------------------------+  15 GB (T4)
   |                                                  |
   |  ACTIVATIONS + framework overhead                |   <- varies
   |  ................................................|
   |                                                  |
   |  KV CACHE                                        |   <- grows with
   |  every token attends to every previous token,    |      CONTEXT LENGTH
   |  so K and V for the whole sequence are cached    |      and BATCH SIZE
   |  ................................................|
   |                                                  |
   |  MODEL WEIGHTS                                   |   <- fixed once
   |  params x bytes-per-weight                       |      loaded
   |                                                  |
   +--------------------------------------------------+

   THE TRAP
   --------
   You test with a 500-token prompt.   Everything fits. Ship it.
   A user pastes a 50-page contract.   KV cache explodes. OOM.

   The weights are constant. The cache is a function of your input,
   which means your memory ceiling is set by your WORST request,
   not your average one.
Why this changes your API design

Because memory scales with context length and concurrency, a self-hosted LLM service must enforce an input token limit at the edge — before the request reaches the model. Rejecting a 200K-token request with a clean 413 is infinitely better than accepting it and crashing the worker for every other user in flight. Same discipline as a max upload size on a file endpoint.

Model weights fit in 4GB of a 15GB GPU. Why can you still run out of memory?

Because weights are the floor, not the ceiling. The KV cache grows with sequence length and with the number of concurrent requests, and activations spike during the forward pass. Long inputs, big batches, or several users at once can consume many times the weight footprint.

It's the classic capacity-planning error: sizing for the static component and forgetting the one that scales with load.

Design question: your Q&A service must serve 50 concurrent users on one GPU. What levers do you have?

Cap input length at the API gateway — the single most effective lever, since KV cache is linear in tokens.
Use a real inference server (vLLM, TGI) rather than a loop over pipeline(). These implement paged attention and continuous batching, which is the difference between 3 concurrent users and 50 on identical hardware.
Choose a model with grouped-query attention — it shrinks the KV cache substantially for the same quality.
Quantize the KV cache, not just the weights.
Queue and admission-control: bound the in-flight requests and make the 51st user wait rather than making all 50 fail.

The meta-point: the notebook in this module can serve exactly one user at a time, and nothing about it degrades gracefully. Concurrency is not a tuning parameter you add later — it's an architectural property of the serving layer you chose.

Part three

Transformers: three levels of control

The library gives you the same capability at three altitudes. Knowing which altitude to fly at — and what each one hides — is most of the skill.

07

Pipeline — the easy door

A pipeline is a ready-made application: it downloads the model and tokenizer, handles preprocessing, runs inference, and converts raw scores back into human-readable output.

Sentiment in three lines — FinBERT on financial text
from transformers import pipeline

clf = pipeline("text-classification", model="ProsusAI/finbert")

print(clf("Tesla stock surges after record-breaking quarterly earnings"))
# [{'label': 'positive', 'score': 0.94}]

print(clf("Microsoft is scheduled to release its earnings report next week"))
# [{'label': 'neutral', 'score': 0.88}]  <- correctly NOT positive
Why the neutral result is the interesting one

A naive keyword classifier sees "earnings report" and guesses positive. FinBERT correctly identifies a scheduling announcement as carrying no sentiment. That distinction — between an event and a judgement about an event — is exactly what domain fine-tuning buys you, and it's why a 110M-parameter specialist can beat a general-purpose giant on a narrow task while costing a thousandth as much to run.

Other task strings worth knowing: text-generation, summarization, translation, question-answering, zero-shot-classification, image-classification, automatic-speech-recognition.

When should you stop using pipeline() and drop a level?

When you need control over something it abstracts away: quantization config at load time, custom device placement, batching strategy, access to logits or attention weights, streaming output token by token, or reusing one loaded model across several different tasks without loading it twice.

Note the useful middle path this module actually uses: load the tokenizer and model yourself with full control, then wrap them in a pipeline. You get the configuration you need and the convenience you want.

Abstractions that hide five steps also hide five failure modes. Which ones does pipeline() hide?

Silent truncation. Input longer than the model's window gets cut, usually without a loud error. Your model answers confidently about a document whose second half it never saw.

Default model selection. pipeline("sentiment-analysis") with no model argument silently picks a small English default — fine in a tutorial, wrong the moment your text is Hindi or financial.

Label semantics. "POSITIVE"/"LABEL_1"/"positive" vary by model; code that string-matches one breaks on another.

Device placement and dtype, chosen for you — which is how you end up running on CPU without noticing.

The general principle: convenience abstractions make the happy path shorter and the failure path more mysterious. Use them, but know exactly what you'd have written by hand.

Models don't read text. They read integers. The tokenizer is the dictionary that converts between the two — and every model family has its own dictionary.

Inspect any tokenizer in four lines
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained("gpt2")
text = "AI is transforming industries and the future of work."

print(tok.tokenize(text))              # human-readable pieces
print(tok(text)["input_ids"])          # what the model actually sees
print(len(tok(text)["input_ids"]))     # your billing unit / context spend
The rule you must not break

The tokenizer must come from the same checkpoint as the model. Always AutoTokenizer.from_pretrained(model_id) with the identical ID you passed to the model. Mismatching them doesn't crash — it silently produces garbage, which is far worse than crashing, because it looks like the model is just bad.

Why tokenization is a business concern, not a detail

  • It's your billing unit. API pricing and context limits are in tokens, not words.
  • It's unfair across languages. Most tokenizers were fitted on English-heavy corpora. The same sentence in Hindi, Tamil or Arabic often costs 2–3× more tokens than in English — meaning higher cost, and less content fitting in the same window. If you're building for Indian users, measure this before you size anything.
  • It explains "dumb" failures. Character-counting and spelling tasks fail because the model never sees characters. Arithmetic is shaky partly because numbers split into arbitrary chunks.
  • It bounds retrieval design. Chunk sizes for RAG are token budgets, so you need the tokenizer to plan them honestly.

Why does mismatching a tokenizer produce fluent nonsense rather than an error?

Because the model receives a perfectly valid sequence of integers. It has no way to know those integers were produced by a different dictionary — they're all inside its vocabulary range, so nothing is out of bounds. It does exactly what it always does: predicts a plausible continuation of the token sequence it was given.

This is a general and important category of bug: a type error that the type system can't see. Both sides agree the payload is "a list of ints"; they disagree on what the ints mean. The same shape hides the same class of bug in currency codes, unit systems and enum mappings.

Your Hindi-language support bot costs 2.5× more per conversation than the English one, on the same model. Explain it to your finance team, then fix it.

The explanation: billing is per token, and tokenizers are fitted to their training corpus. A vocabulary learned mostly from English represents common English words as single tokens but shatters Devanagari into many small pieces. Same meaning, more tokens, more cost — and less conversation fitting in the context window before truncation.

The fixes, in order: pick a model whose tokenizer was trained multilingually (several Qwen and Gemma builds handle Indic scripts far more efficiently); measure tokens-per-message empirically across your actual traffic rather than assuming; shorten system prompts, which you pay for on every single turn; and consider summarising conversation history rather than resending it.

The wider point for anyone building for Indian users: tokenizer efficiency is a first-class model selection criterion here, and it never appears on a leaderboard.

AutoModelForCausalLM — loading the brain

Causal means left-to-right: the model predicts the next token given everything before it, and can never look ahead. That's the architecture behind every generative chat model.

              |
              v
   +----------------------------+
   | forward pass through model |
   +----------------------------+
              |
              v   next-token probabilities
        Paris 0.91   the 0.03   a 0.02  ...
              |

Your security team asks whether downloading models from Hugging Face is different from installing a Python package. Answer them.

It's the same category of risk with a larger and less inspectable payload. A pip package is code you can read; a model is gigabytes of opaque tensors plus, sometimes, code you can read.

Comparable risks: arbitrary code execution on load (pickle-format checkpoints, trust_remote_code), typosquatted repositories, unpinned versions changing under you.

Risks with no packaging analogue: the weights themselves can be backdoored — trained to behave normally except on a trigger phrase — and no static scan will find that. You cannot diff a tensor for intent.

Therefore the controls are: safetensors only; pin revisions; mirror into an internal registry after scanning; treat any model touching a sensitive path as requiring behavioural evaluation, not just a file scan. And be honest that the last control is imperfect — which is itself an argument for preferring a small number of models from major labs over a long tail of community fine-tunes.

The reasoning behind temperature 0.2 here

In Section 5 you wanted variety — a poem, a landing page, several plausible designs. Here you want the opposite: the answer that is actually in the document, reproducibly. High temperature in a grounded Q&A system doesn't make it creative, it makes it wrong more often — and in a financial context, a confidently-worded invented number is the single most dangerous output your system can produce.

Match temperature to consequence: creative tasks tolerate variance, extraction tasks do not.

Same prompt, same model, different answer each run. Is that a bug?

Not by itself — it's do_sample=True doing exactly what it's told. Set do_sample=False (greedy decoding) and you get near-determinism for the same input.

But "near". Even greedy decoding can vary slightly across runs due to floating-point non-determinism in GPU kernels and batching effects. If your system requires byte-identical reproducibility — an audit trail, a regression test — you cannot get it from the model alone. You cache the output and audit the cache. That's a real architectural constraint people discover late.

Latency is proportional to output length, not input length. Why, and what does that imply for your UX?

The prompt is processed in a single parallel forward pass (prefill), but every output token requires its own sequential pass. 500 output tokens means 500 round trips through the network, one after another, that cannot be parallelised.

UX implications: stream tokens as they're produced, so perceived latency is time-to-first-token rather than total time. Ask for concise answers in the prompt — brevity is a latency optimisation, not just a style preference. And cap max_new_tokens to what you'll actually display; generating text nobody reads costs real seconds and real VRAM.

It also explains a counterintuitive result: feeding a much longer document often costs less added time than asking for a slightly longer answer.

When pypdf isn't enough

  • Complex layouts → layout-aware parsers that preserve reading order and hierarchy.
  • Any of the above at scale → an ingestion pipeline with per-document quality checks, not a for-loop.

Write three automated checks your ingestion step should run on every extracted document.

1. Non-empty and plausible density. Characters-per-page below some floor almost certainly means a scanned or image-based page — flag for OCR rather than passing an empty string downstream.

2. Page-count reconciliation. Number of extracted page-strings equals len(reader.pages), and no page is silently empty in the middle of a run.

3. Encoding and content sanity. Check for a reasonable ratio of alphabetic characters — a wall of replacement characters or ligature artefacts means the font mapping failed.

A fourth, if figures matter: verify that a known anchor string (a heading, a date, a total you expect) is present. Cheap, and it catches the scrambled-column case that the other three miss.

Why is document ingestion, not the model, usually the biggest quality lever in a real RAG system?

Because everything downstream is bounded by it. A frontier model reasoning over scrambled text produces confident wrong answers; a modest model reasoning over cleanly parsed, well-chunked text produces good ones. You cannot prompt your way out of corrupted input.

Teams consistently misallocate here — they spend weeks comparing models and an afternoon on the parser, then wonder why accuracy plateaus. In most document-AI projects the ranking of effort should be: ingestion quality > chunking and retrieval > prompt design > model choice. The last one gets all the attention because it's the most fun and the easiest to have opinions about.

      An instruction is a request, not a constraint

      Your model answers a question correctly, but the answer is nowhere in the uploaded document. Bug or feature?
      

        
Bug — and a dangerous one, because correctness masks it. The system's contract is "answer from this document." An answer sourced from training data is unverifiable, uncitable, possibly outdated, and will be confidently wrong the day the document contradicts what the model memorised.

        
Worse, it makes your testing misleading. You test on Google's earnings call, the model already knows Google's numbers, everything looks excellent. Then you deploy on your own internal documents that the model has never seen, and accuracy falls off a cliff.       Why does giving the model an explicit way to say "I don't know" improve accuracy more than telling it not to hallucinate?
      

        
Because the model is fundamentally a next-token predictor optimised to produce a plausible continuation. "Don't hallucinate" is a negative instruction with no alternative behaviour attached — the model still has to emit something, and something plausible is exactly what hallucination is.

        
Giving it a specific, high-probability output for the not-found case ("The document does not contain information on that topic") supplies a concrete continuation that competes with invention. You've turned an abstract prohibition into a concrete choice between two available paths.

        
This generalises well beyond LLMs: specifying the desired fallback behaviour beats prohibiting the undesired one. Same reason error handling works better than "don't crash."

      

    


    

      You support four models behind a dropdown. Where does the chat-template logic belong in your codebase?
      

        
Nowhere in your codebase — that's the point. Templates belong to the model, so tokenizer.apply_chat_template() is the abstraction boundary. Adding a fifth model becomes a config entry, not a code change.

        
If you hand-write templates instead, you've created a lookup table that must be updated for every new model, silently rots when a vendor revises their format, and produces degraded quality rather than a loud error when it's wrong. That's the worst failure signature possible: a config bug that manifests as "the model seems dumber than the benchmarks said."

        
Generalisable rule: when a vendor ships metadata describing how to talk to their artefact, consume the metadata rather than re-encoding your interpretation of it.

  • Chunk and score by keyword overlap. Split into ~1,000-character chunks, rank them by term overlap with the question (BM25 is a short function), send the top 5. No embeddings, no new infrastructure, dramatically better coverage.

  • Truncate in the middle, not the end. Documents front-load context and back-load conclusions; keeping the head and tail beats keeping only the head.

      Why is truncating by characters rather than tokens a second, separate bug?
      

        
Because the model's limit is in tokens, and the characters-to-tokens ratio varies by content and language. 6,000 characters of English prose is roughly 1,500 tokens; 6,000 characters of dense tables, code, or Devanagari can be two to three times that. You've set a limit in a unit that doesn't correspond to the constraint you're managing.

        
Consequence: your safety margin is illusory. It holds for the document you tested and silently blows through the context window on a different one — which then gets truncated again, invisibly, inside the tokenizer. Measure in the unit the system actually enforces.

      

    


    

      Your team wants document Q&A over 10,000 internal PDFs. Sketch the architecture, and name the parts this notebook doesn't have.
      

        
Ingestion: a pipeline that parses (with OCR fallback), validates quality, extracts metadata (source, page, date, access level), chunks with overlap, and is re-runnable when the parser improves.

        
Indexing: an embedding model, a vector store, and — importantly — hybrid search, because pure semantic retrieval is poor at exact identifiers like invoice numbers and ticker symbols.

        
Retrieval: top-k with a reranker, plus permission filtering applied at query time. This is the one most prototypes get catastrophically wrong: if retrieval ignores access control, your Q&A bot becomes a data-exfiltration tool that cheerfully summarises documents the user was never allowed to open.

        
Generation: the model, with mandatory citations back to chunk and page.

        
Evaluation: a labelled question set with known answers, measuring retrieval hit-rate and answer faithfulness separately — because when quality drops you must know which half broke.

        
Operations: incremental re-indexing, cost per query, latency budget, an audit log of who asked what and which documents were surfaced.

        
This notebook has the generation step and nothing else. That's not a criticism of the notebook — it's a map of the work.

      

    


    

      RAG adds retrieval failure modes on top of generation failure modes. Name two, and say how you'd detect them.
      

        
Retrieval miss: the correct chunk exists but wasn't retrieved, so the model correctly says "not in the document" — a false negative that looks like polite, well-behaved output. Detect by measuring retrieval recall separately against a labelled set: for each question, is the gold chunk in the top-k? An answer-only metric will never surface this.

        
Distractor retrieval: chunks that are semantically similar but factually about a different quarter, entity or year get retrieved, and the model dutifully answers from the wrong one. This is worse than a miss because the output is confident and specific. Detect by requiring citations and auditing whether the cited chunk actually supports the claim — which is exactly why citations are a correctness mechanism, not a UI nicety.

        
The general lesson: in a multi-stage pipeline, end-to-end metrics tell you that something is broken and never which thing. Instrument each stage.

The unload sequence — order matters


import gc, torch

CURRENT = {"id": None, "pipe": None, "model": None, "tok": None}

def load_llm_model(display_name):
    model_id = AVAILABLE_MODELS[display_name]
    if model_id == CURRENT["id"]:
        return f"{display_name} already loaded."

    # --- unload: all three steps, in this order ---
    for k in ("pipe", "model", "tok"):
        CURRENT[k] = None          # 1. drop Python references
    gc.collect()                     # 2. actually collect them
    torch.cuda.empty_cache()         # 3. return the VRAM to the driver

    # --- load the new one ---
    tok = AutoTokenizer.from_pretrained(model_id)
    model = AutoModelForCausalLM.from_pretrained(
        model_id, quantization_config=bnb, device_map="auto")
    pipe = pipeline("text-generation", model=model, tokenizer=tok)

    CURRENT.update(id=model_id, pipe=pipe, model=model, tok=tok)
    return f"{display_name} loaded."

    


    

      Why del on its own does nothing
      
Python's del removes a name binding. The GPU memory is only released when the last reference is gone and the garbage collector runs and PyTorch's caching allocator returns the block. Miss any of the three and your "unloaded" model is still occupying VRAM — which is why the second model swap OOMs even though the first worked. This exact sequence is the fix.

    


    

      Also note: the dropdown label mapping is a real design choice
      
Mapping "Llama 3.2 (3B)" → "meta-llama/Llama-3.2-3B-Instruct" in a dictionary isn't cosmetic. It decouples the UI from the model registry, so swapping a checkpoint doesn't change what users see, and adding a model is a one-line config change. Small pattern, but it's the difference between a demo and something maintainable.

    


    

      Gradio docs
    


    

      Two users hit this Gradio app at once and one switches models. What happens?
      

        
Chaos. CURRENT is process-global with no locking. User A's question can be answered by user B's model, or by a half-loaded one; worse, a swap triggered mid-generation frees memory a running forward pass is still using.

        
This isn't a Gradio flaw — it's an architecture that assumes a single user, made explicit. The fix isn't a mutex bolted on; it's recognising that model lifecycle and request handling belong in different layers. Real systems load models into dedicated long-lived workers and route requests to them; they never load a model in response to a UI event.

      

    


    

      Your demo works. Someone asks "can we put this in front of customers next month?" What do you say?
      

        
Say yes to the idea and be precise about the gap. Missing pieces, roughly in order of how badly they bite:

        
Concurrency — one global model, no queue, no locking. Two users break it.

        Cold starts — a model swap takes minutes and downloads gigabytes. No customer waits.

        Coverage — 6,000 characters of a 64,000-character document. Retrieval is not optional.

        Groundedness — no citations, no verification that answers came from the source.

        Access control — no notion of who may see which document.

        Evaluation — no labelled question set, so no way to detect regression when anything changes.

        Observability — no logging of questions, retrieved context, latency or cost per query.

        Failure behaviour — an OOM takes down the whole app for everyone.

        
The framing that lands with leadership: the notebook proved feasibility, which was the risky unknown. What remains is engineering with known solutions and a knowable cost. That's a good position to be in — just not a one-month one.

"""Open-weight document Q&A. Colab: pip install -q transformers accelerate
bitsandbytes torch pypdf gradio"""
import gc, re, torch, gradio as gr
from pypdf import PdfReader
from transformers import (AutoTokenizer, AutoModelForCausalLM,
                          BitsAndBytesConfig, pipeline)

# ---------- 1. model registry: UI label -> hub id -----------------
MODELS = {
    "Phi-3 mini (3.8B)":  "microsoft/Phi-3-mini-4k-instruct",
    "Llama 3.2 (3B)":     "meta-llama/Llama-3.2-3B-Instruct",
    "Qwen 2.5 (3B)":      "Qwen/Qwen2.5-3B-Instruct",
    "Gemma 3 (4B)":       "google/gemma-3-4b-it",
}
BNB = BitsAndBytesConfig(load_in_4bit=True,
                         bnb_4bit_quant_type="nf4",
                         bnb_4bit_compute_dtype=torch.float16,
                         bnb_4bit_use_double_quant=True)
CUR = {"id": None, "pipe": None, "model": None, "tok": None}

# ---------- 2. PDF ingestion with a real sanity check -------------
def read_pdf(path):
    reader = PdfReader(path)
    pages = [(p.extract_text() or "") for p in reader.pages]
    empty = [i for i, t in enumerate(pages) if len(t.strip()) < 50]
    if empty:
        print(f"WARNING: pages {empty} look empty — scanned? needs OCR")
    return pages

# ---------- 3. retrieval-lite: rank chunks, don't truncate --------
def best_chunks(pages, question, size=1200, overlap=200, k=5):
    """Keyword overlap beats [:6000] and needs no vector DB."""
    chunks = []
    for pno, text in enumerate(pages, start=1):
        for i in range(0, len(text), size - overlap):
            body = text[i:i + size].strip()
            if body:
                chunks.append((pno, body))

    q = set(re.findall(r"\w+", question.lower())) - {
        "the", "a", "of", "is", "in", "what", "does", "and"}
    scored = sorted(chunks,
        key=lambda c: len(q & set(re.findall(r"\w+", c[1].lower()))),
        reverse=True)[:k]
    scored.sort(key=lambda c: c[0])                # restore doc order
    return "\n\n".join(f"[page {p}]\n{t}" for p, t in scored)

# ---------- 4. model lifecycle ------------------------------------
def load_model(label):
    mid = MODELS[label]
    if mid == CUR["id"]:
        return f"{label} already loaded."
    for key in ("pipe", "model", "tok"):
        CUR[key] = None
    gc.collect(); torch.cuda.empty_cache()

    tok = AutoTokenizer.from_pretrained(mid)
    mdl = AutoModelForCausalLM.from_pretrained(
        mid, quantization_config=BNB, device_map="auto")
    CUR.update(id=mid, tok=tok, model=mdl,
               pipe=pipeline("text-generation", model=mdl, tokenizer=tok))
    return f"{label} loaded."

# ---------- 5. the grounded answer --------------------------------
SYSTEM = ("You answer questions about a document. Use ONLY the document "
          "text below. If the answer is not there, reply exactly: "
          "'The document does not contain information on that topic.' "
          "Cite the page number you used. Do not use prior knowledge.")

def answer(pages, question):
    if CUR["pipe"] is None:
        return "Load a model first."
    context = best_chunks(pages, question)
    messages = [
        {"role": "system", "content": f"{SYSTEM}\n\nDOCUMENT:\n{context}"},
        {"role": "user",   "content": question},
    ]
    prompt = CUR["tok"].apply_chat_template(
        messages, tokenize=False, add_generation_prompt=True)
    out = CUR["pipe"](prompt, max_new_tokens=400, do_sample=True,
                       temperature=0.2, top_p=0.9,
                       return_full_text=False)          # no string slicing
    return out[0]["generated_text"].strip() or "(empty response)"

# ---------- 6. UI --------------------------------------------------
PAGES = read_pdf("earnings_call.pdf")

with gr.Blocks() as demo:
    gr.Markdown("# PDF Q&A — open-weight models")
    pick   = gr.Dropdown(list(MODELS), value=list(MODELS)[0],
                         label="Model")
    status = gr.Textbox(label="Status", interactive=False)
    q      = gr.Textbox(label="Your question")
    btn    = gr.Button("Ask", variant="primary")
    ans    = gr.Textbox(label="Answer", lines=8)

    pick.change(load_model, inputs=pick, outputs=status)
    btn.click(lambda question: answer(PAGES, question),
              inputs=q, outputs=ans)

demo.launch(debug=True)

    


    

      Four changes from the tutorial, and why
      
1. apply_chat_template instead of hand-written <|system|> tokens — so all four models work without per-model formatting code.

      2. best_chunks instead of [:6000] — the whole document becomes answerable with no new infrastructure.

      3. return_full_text=False instead of splitting on "Answer:" — a structural boundary rather than a content guess.

      4. Page numbers in the context and a citation instruction — so answers can be checked.

  ================================================================
   THE MEMORY FORMULA  (memorise this one)
  ================================================================
   VRAM for weights (GB)  ~=  params(B)  x  bytes-per-weight
     fp32 = 4      fp16/bf16 = 2      int8 = 1      int4 = 0.5

     7B fp16 = 14 GB  -> will NOT fit a 15GB T4
     7B int4 = 3.5 GB -> fits easily

   ...then add KV cache, which grows with CONTEXT x BATCH.
   Weights are the floor. Your ceiling is set by your WORST request.

  ================================================================
   THREE LEVELS OF CONTROL
  ================================================================
   pipeline("task")           easiest. hides truncation, defaults,
                              label naming, device placement
   AutoTokenizer +            the working level: you control
   AutoModelForCausalLM       quantization + placement, then wrap
     -> then pipeline(...)    them in a pipeline
   raw generate()             when you need logits or streaming

  ================================================================
   THE THREE-STEP LOAD
  ================================================================
   model_id  = "org/Model-size-context-instruct"
   tokenizer = AutoTokenizer.from_pretrained(same id)
   model     = AutoModelForCausalLM.from_pretrained(
                   model_id, quantization_config=bnb,
                   device_map="auto")

   NEVER mix a tokenizer from one model with weights from another.
   It does not error. It produces fluent nonsense.

  ================================================================
   PROMPTING AN OPEN MODEL
  ================================================================
   messages = [{"role":"system",...},{"role":"user",...}]
   prompt = tokenizer.apply_chat_template(
              messages, tokenize=False, add_generation_prompt=True)

   Do NOT hand-write <|system|> / <|im_start|> / <start_of_turn>.
   Every family differs; the tokenizer already knows.

   Grounding recipe: "answer ONLY from the document; if it isn't
   there say <exact string>; cite the page; no prior knowledge."
   -> give the model a licensed way to FAIL, or it will invent.

  ================================================================
   DECODING SETTINGS
  ================================================================
   document Q&A ->  temperature 0.1-0.2, top_p 0.9, max_new 300-500
   creative     ->  temperature 0.7-1.0
   latency scales with OUTPUT length, not input length -> stream

  ================================================================
   THE BIG ARCHITECTURAL TRUTH
  ================================================================
   context = document[:6000]      <-- this is not a system.
                                      it is a placeholder for
                                      CHUNK -> EMBED -> RETRIEVE
                                      -> RERANK -> CITE

   In document AI, effort should go:
     ingestion quality > retrieval > prompt > model choice
   Most teams do it exactly backwards.

  ================================================================
   GOTCHAS
  ================================================================
   * del alone does NOT free VRAM. del + gc.collect()
     + torch.cuda.empty_cache() - all three, in that order
   * device_map="auto" can silently offload to CPU -> 50x slower.
     check model.hf_device_map
   * base vs instruct: base models do not follow instructions
   * trust_remote_code=True runs their Python on your machine
   * pypdf silently returns "" for scanned pages, and scrambles
     multi-column layouts and tables
   * truncating by CHARACTERS when the limit is in TOKENS
   * Colab idles out and wipes everything. That is not a bug,
     it is a reminder that a notebook is not infrastructure.

Glossary

    

      
Open weights
Downloadable model parameters you can run, inspect and fine-tune. Not the same as open source, which would also mean public training data and code.

      
Hugging Face Hub
Public registry of models, datasets and demo Spaces. The npm of machine learning.

      
Model card
The README of a model: licence, size, context, intended use, benchmarks, limitations. Read the licence before the benchmarks.

      
Gated model
Requires accepting terms and authenticating with a token before download. Llama models are gated.

      
Base vs Instruct
Base models continue text. Instruct/it/chat models have been further trained to follow requests. Use instruct for anything conversational.

      
Parameters (B)
Billions of learned weights. Predicts VRAM need, latency and roughly capability.

      
Quantization
Storing weights at lower precision (int8, int4) to shrink memory. Lossy, but neural networks tolerate it remarkably well.

      
NF4
A 4-bit format designed for the bell-shaped distribution of neural network weights. Better quality than naive int4 at the same size.

      
bitsandbytes
The library that performs on-the-fly quantization when loading a model in Transformers.

      
VRAM
GPU memory. The binding constraint in self-hosting — capacity stops you before speed does.

      
KV cache
Cached attention keys and values for the sequence so far. Grows with context length and batch size; the hidden memory cost.

      
Transformers
Hugging Face's Python library for loading and running models. Built on the 2017 "Attention Is All You Need" architecture.

      
Pipeline
High-level wrapper that chains tokenizer → model → post-processing into one call.

      
Tokenizer
Converts text to integer IDs and back. Model-specific — must always match the checkpoint.

      
Token
The unit a model reads. Roughly ¾ of an English word; considerably less efficient for Indic scripts.

      
Causal LM
Left-to-right next-token prediction. The architecture behind all generative chat models.

      
Chat template
The exact special-token format a model was fine-tuned on. Retrieve it with apply_chat_template; never hand-write it.

      
Special tokens
Markers like <|user|> or <|im_start|> that tell the model whose turn it is.

      
device_map="auto"
Lets accelerate place layers across GPU, CPU and disk. Convenient, and a silent performance trap when VRAM is short.

      
trust_remote_code
Executes custom Python shipped in the model repo. A security decision, not a convenience flag.

      
safetensors
A checkpoint format that cannot execute code on load, unlike legacy pickle files. Always prefer it.

      
Grounding
Constraining answers to a supplied source rather than the model's training data.

      
RAG
Retrieval-Augmented Generation: chunk, embed, retrieve the relevant pieces, then generate from them. What [:6000] is standing in for.

      
Chunking
Splitting a document into overlapping pieces sized to your context budget. Overlap prevents answers being severed at boundaries.

      
Continuous batching
Serving technique (vLLM, TGI) that packs requests together dynamically. The difference between 3 and 50 concurrent users on one GPU.

      
Gradio
Python library for building quick web UIs around models. Excellent for demos; not a production serving layer.

      
OCR
Optical Character Recognition — needed when a PDF is a scanned image and extract_text() returns nothing.
