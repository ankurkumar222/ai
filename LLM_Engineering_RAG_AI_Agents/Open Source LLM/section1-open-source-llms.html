Part one

The open-weight decision

Before any code: what changes when the model sits on your disk instead of someone else's server. This looks like a technical choice. It is really a business one.

01

What changes when you own the weights

In Section 5 you called GPT-4o, Gemini and Claude. You sent text, you got text back, and you never touched the model. Here you download the model itself — billions of numbers — and run it on hardware you look after.

The five real reasons to self-host

01 · Data residency

Nothing leaves the perimeter

The big one in banking, healthcare and defence. If customer data or KYC documents are not allowed to leave your network, it does not matter which model scores higher. Running it yourself is the only legal option.

02 · Unit economics at volume

Fixed cost beats per-token

With an API you pay per use, so the bill grows with traffic. A GPU costs the same whether you use it or not. Below a certain volume the API is cheaper. Above it, your own GPU is. Find that line before you argue about it.

03 · Customisation

Fine-tuning is only possible here

You can keep training the model on your own data — your product catalogue, your regulatory wording, your support tickets. No closed API lets you change the model itself like this.

04 · Version stability

The model doesn't change under you

A model you downloaded never changes. Hosted models get updated or retired without warning, which can break prompts you spent weeks tuning. If you need to reproduce a result later, this matters a lot.

05 · Offline & edge

Works with no internet

Isolated networks, factory floors, phones, backup sites. An API simply cannot work here.

Bonus · Transparency

You can inspect it

You can see how the model is built. Useful for research, for security reviews, and for answering a regulator who asks how it works.

The honest counter-argument

Most teams underestimate the real cost. You now pay for: a GPU sitting idle between requests, a server to run the model on, re-testing every time you change models, monitoring, a security review of what you downloaded, and engineers who can debug GPU errors at 2am. A top API is usually cheaper and better until you hit a legal wall or very high volume. "Let's self-host to save money" is the most common wrong reason.

Name the one reason to self-host that no amount of API improvement can ever overturn.

The legal one — data cannot leave your network. Arguments about cost, quality and speed all change as vendors improve and prices drop. But if the law or your contract says the data must stay inside, no API becomes allowed, however cheap or good it gets. Every other reason moves. This one does not.

Your CFO asks: "at what point does self-hosting actually become cheaper?" Sketch how you'd answer.

Give them a number, not an opinion. API cost = tokens per request × requests per month × price per token. Self-host cost = GPU hours per month + engineering time + storage. Then find the traffic level where the two are equal.

Two things people forget. Idle time: a GPU you rent all day but use 6% of the time makes your real cost per answer huge. With an API you only pay when you use it, and that is worth something. Retries: if the open model needs three attempts to match one answer from a top model, your token count is wrong.

The honest answer is usually: "cheaper above roughly X million tokens a month if the GPU stays busy, and today we are at 5% of X." That moves the conversation to the reason that actually applies — rules or customisation.

Frozen weights are listed above as an advantage. When are they a serious liability?

The moment everyone else's model gets better and yours does not. A frozen model keeps every bias, every security hole and every factual error it shipped with, forever. Hosted models get fixed for you. Yours does not, unless you download a new one and test it again.

So "stable version" is not free. It moves a job the vendor used to do onto your team. If you self-host and never look at the model again, you have not chosen stability — you have chosen slow decay. The grown-up version is a planned upgrade rhythm: fixed version in production, reviewed every quarter against your own test set.

You load a model and it rambles instead of answering your question. What's the first thing to check?

Whether you downloaded the base model instead of the instruct one. Base models only know how to continue text. Ask one "What is the capital of France?" and it may happily write five more exam questions instead of answering. The instruct versions (-instruct, -it, -chat) have been trained to treat your input as a request.

Second thing to check: whether you applied the model's chat template. Right model, wrong formatting produces the same symptom.

Why is "1.6 million models available" a problem as much as a feature?

Because now choosing is the hard part. Most of those models are half-finished experiments and abandoned copies. Anyone can upload, so you get huge choice and no quality signal at all.

The fix is to narrow down hard before you browse. Filter by task, then by an organisation you trust to still exist next year (Meta, Google, Microsoft, Alibaba, Mistral, DeepSeek), then by licence, then by a size that fits your GPU. That turns 1.6 million into about a dozen. Pick models the way you pick libraries. You would not drop a random package with four stars into a payments system, and a model is a much bigger act of trust than a package.

What is the supply-chain security question nobody in the tutorial asks?

You are downloading files from a public site and loading them into a program that can reach your GPU and your disk. A security review will ask:

File format: is it safetensors or the old pickle format? Pickle files can run code when you open them. Always prefer safetensors.
Custom code: does it need trust_remote_code=True? That means "run this repo's Python on my machine" — see section 09.
Who published it: is this the official account or a copycat with a similar name? Fake lookalikes exist here too.
Version: are you pulling the latest, or a fixed version? If you do not fix it, the file can change under you.
Copy it once: in a regulated company, download the model once, scan it, and serve it from your own internal store — never pull it from the internet every time a container starts.

The parameter-count intuition

B = billions of weights. Qwen3-32B holds 32 billion numbers. This one figure tells you your GPU cost, your speed and roughly how capable it is — which is why it is in the name. A simple ladder: 0.5–3B runs on a laptop or a free Colab GPU and handles extraction, sorting and simple Q&A. 7–14B is the sweet spot for most real self-hosted work. 30B+ needs serious hardware and starts to feel like a top model on narrow tasks.

Two candidates: Phi-3-mini-4k-instruct and Phi-3-mini-128k-instruct. Same family, same size. When does the 4k version still win?

When your inputs are short. A longer context is not free. It uses more GPU memory (section 06), it is slower, and long-context versions are sometimes slightly weaker on short inputs. If you are sorting 200-token support tickets, a 128k window is renting a warehouse to store one envelope.

The 128k version wins the moment a document genuinely cannot be split up without losing its meaning.

You see -GGUF and -AWQ variants of a model you like. What does their existence tell you about the model's real-world usage?

That real people are running it on small hardware. Those are community-made compressed versions, and they only exist because someone wanted them. So their presence is a good sign that the model has a healthy community: tools that support it, and people who have already hit the bugs and written them down.

It also saves you work. If a good compressed version already exists, you may not need to compress the model yourself when loading. You can download one someone has already tested, which loads faster and is often better tuned than doing it on the fly.

Part two

Hardware is the real constraint

With APIs, the limit was how smart the model is. Here the limit is memory. This part is the gap between the model you want and the model you can actually run. Most self-hosting projects fail right here.

04

Why a GPU, and which one

Running an LLM is basically a huge pile of multiplication. Every word it produces means pushing numbers through billions of weights. A CPU does a few of these at a time, very fast. A GPU does thousands at once, each a bit slower. For this kind of work, doing many at once wins by a mile.

   [====>]                             [=>][=>][=>][=>][=>][=>][=>][=>]

VRAM matters more than speed

Watch out for this trap. People compare GPUs on speed, but the thing that actually blocks you is space. A slow GPU with enough memory will run your model. A fast GPU without enough memory throws CUDA out of memory and runs nothing at all. Check that the model fits first. Worry about speed second.

In Colab: Runtime → Change runtime type → Hardware accelerator. Then verify from code rather than trusting the menu:

Always verify the GPU before loading anything
import torch

if torch.cuda.is_available():
    name = torch.cuda.get_device_name(0)
    total = torch.cuda.get_device_properties(0).total_memory / 1024**3
    print(f"GPU: {name} — {total:.1f} GB VRAM")
else:
    print("No GPU. Loading a 3B model here will take minutes per token.")
Your notebook idles for twenty minutes, then every cell fails with "torch is not defined". What happened?

Colab took your machine back. Everything is gone: imports, variables, the downloaded model, your login. You have to run from the top again — install, import, log in, reload the model, which downloads gigabytes all over again.

The lesson underneath the annoyance: a notebook is not real infrastructure. Anything that takes five minutes to rebuild and only lives in memory is a demo, not a system. Real deployments load the model once into a long-running service exactly to avoid this.

Why would a production team accept a slower GPU than the one their benchmark says is fastest?

Because what drives cost is how busy the GPU is, not how fast it is. A cheap GPU that is busy 70% of the time beats an expensive one that is busy 8% of the time, every time. On top of that, the best GPUs are often simply unavailable in your region, and a bigger card only helps if your model actually fills it.

The real question is never "which GPU is fastest?" It is "what does a thousand answers cost me at real traffic, counting idle time?" Asked that way, the answer is usually a smaller card, better batching, and scaling down to zero when nobody is using it — or, honestly, going back to the API.

05

Precision and quantization

Every weight is a number, and you choose how many bits to store it in. That one choice decides more than anything else whether the model fits on your hardware.

Sketch 5 · The precision ladder
   FORMAT   BITS  BYTES/WEIGHT   7B MODEL   WHAT YOU GIVE UP
   ------------------------------------------------------------------
   fp32      32      4.0          ~26 GB    nothing (training default)
   fp16/bf16 16      2.0          ~13 GB    almost nothing for inference
   int8       8      1.0          ~6.5 GB   very little
   int4       4      0.5          ~3.5 GB   a little; usually acceptable
                                            ^^^^^^^^^^^^^^^^^^^^^^^^^
   THE FORMULA YOU SHOULD MEMORISE
   -------------------------------------------------------------
       VRAM for weights (GB)  ~=  params(B)  x  bytes-per-weight

       7B  x 2 bytes (fp16) = 14 GB   -> will NOT fit a 15GB T4 with
                                         room for anything else
       7B  x 0.5 (int4)     = 3.5 GB  -> fits comfortably

   Quantization is lossy compression for weights. You are rounding
   billions of numbers to a coarser grid. The remarkable empirical
   finding is how little that costs you at 8 and even 4 bits.

How you actually ask for it

4-bit quantization via bitsandbytes
from transformers import AutoModelForCausalLM, BitsAndBytesConfig
import torch

bnb = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",              # NF4 > plain int4 for LLM weights
    bnb_4bit_compute_dtype=torch.float16,   # maths still runs in fp16
    bnb_4bit_use_double_quant=True,        # quantize the quantization constants too
)

model = AutoModelForCausalLM.from_pretrained(
    model_id,
    quantization_config=bnb,
    device_map="auto",
)
The subtlety worth understanding

How you store a number and how you calculate with it are two different things. The weights sit in memory at 4 bits, but they are expanded back to 16 bits to do the actual maths. That is what bnb_4bit_compute_dtype is for. It also explains something people find odd: 4-bit saves a huge amount of memory but does not make the model much faster. You shrank what you store, not the work you do. Quantization buys you the ability to run the model at all — not speed.

You quantize to 4-bit and quality drops noticeably on your task. What are your options, in order of cost?

1. Move up to 8-bit. Uses about twice the memory, but usually gets most of the quality back. Cheapest fix if it fits.

2. Use a better compression method. NF4 with double quantization, or a community-made AWQ/GPTQ version that was tuned properly, usually beats a quick conversion at the same size.

3. Use a smaller model at higher precision. A 3B model at 16-bit often beats a 7B squashed to 4-bit — same memory, better answers. People forget this option.

4. Rent a bigger GPU. The honest last resort, and sometimes the right answer.

Quantization is lossy. Why doesn't it destroy the model the way lossy compression destroys a spreadsheet?

Because a neural network has enormous overlap built in, and it is used to noise. No single weight decides anything. The behaviour comes from patterns across billions of them. Tiny rounding errors spread across all of them mostly cancel each other out, and the model was trained with random noise anyway, so it copes with a bit more.

A spreadsheet is the opposite. Every cell matters on its own, and one wrong digit is one wrong answer. That overlap is what makes squashing a model survivable, and it is why the same trick fails anywhere each value stands alone.

Worth sitting with: the damage is not spread evenly. It shows up first in the rarest and most exact skills — long arithmetic, obscure facts, less common languages — while normal conversation still sounds fine. So a squashed model can pass casual testing and then fail on exactly the cases you care about.

When is running a quantized model actively the wrong call, even though it fits?

When failures are silent and the stakes are high. Compression hurts exact work first — maths, pulling out precise figures, rare words. In a system that reads numbers out of financial filings, a 4-bit model that swaps two digits gives you a fluent, confident, wrong answer that no error check will catch.

The test is not "does it still sound good." It is "does it still get the numbers right on my own list of known answers." If you have not measured that, you do not know what the compression cost you. You only know the model loads.

06

The memory budget nobody warns you about

Everyone quotes the weight size. But weights are not the only thing in GPU memory, and the other pieces are exactly what make an app work in testing and fail in production.

Sketch 6 · What's actually sitting in your GPU
   +--------------------------------------------------+  15 GB (T4)
   |                                                  |
   |  ACTIVATIONS + framework overhead                |   <- varies
   |  ................................................|
   |                                                  |
   |  KV CACHE                                        |   <- grows with
   |  every token attends to every previous token,    |      CONTEXT LENGTH
   |  so K and V for the whole sequence are cached    |      and BATCH SIZE
   |  ................................................|
   |                                                  |
   |  MODEL WEIGHTS                                   |   <- fixed once
   |  params x bytes-per-weight                       |      loaded
   |                                                  |
   +--------------------------------------------------+

   THE TRAP
   --------
   You test with a 500-token prompt.   Everything fits. Ship it.
   A user pastes a 50-page contract.   KV cache explodes. OOM.

   The weights are constant. The cache is a function of your input,
   which means your memory ceiling is set by your WORST request,
   not your average one.
Why this changes your API design

Memory grows with how long the input is and how many people are using it at once. So a self-hosted model service must cap the input size at the front door, before the request reaches the model. Politely rejecting a huge request is far better than accepting it and crashing the server for everyone else. It is the same rule as a maximum file upload size.

Model weights fit in 4GB of a 15GB GPU. Why can you still run out of memory?

Because the weights are the floor, not the ceiling. The cache grows with the length of the input and the number of people using it at once, and there is extra working memory during each pass. Long documents, big batches or a few users together can use several times the weight size.

It is the classic planning mistake: you size for the fixed part and forget the part that grows with traffic.

Design question: your Q&A service must serve 50 concurrent users on one GPU. What levers do you have?

Cap the input size at the front door. This is the biggest single win, because cache use grows directly with input length.
Use a proper model server (vLLM, TGI) instead of calling pipeline() in a loop. These pack requests together cleverly, which is the difference between 3 users and 50 on the same hardware.
Pick a model designed with a smaller cache (grouped-query attention) — same quality, much less memory.
Compress the cache too, not only the weights.
Add a queue: limit how many requests run at once and make the 51st person wait, instead of letting all 50 fail.

The bigger point: the notebook in this module serves exactly one person at a time, and it fails hard rather than slowing down. Handling many users is not a setting you turn on later. It comes from the serving layer you picked at the start.

Transformers: three levels of control

The library lets you work at three levels. Knowing which level to use, and what each one hides from you, is most of the skill.

07

Pipeline — the easy door

A pipeline is a ready-made mini app. It downloads the model and tokenizer, prepares your text, runs the model, and turns the raw numbers back into readable output.

Sentiment in three lines — FinBERT on financial text
from transformers import pipeline

clf = pipeline("text-classification", model="ProsusAI/finbert")

print(clf("Tesla stock surges after record-breaking quarterly earnings"))
# [{'label': 'positive', 'score': 0.94}]

print(clf("Microsoft is scheduled to release its earnings report next week"))
# [{'label': 'neutral', 'score': 0.88}]  <- correctly NOT positive
Why the neutral result is the interesting one

A simple keyword checker sees "earnings report" and guesses positive. FinBERT correctly sees that announcing a date carries no opinion at all. Telling the difference between an event and an opinion about an event is exactly what training on one domain buys you. It is why a tiny specialist model can beat a giant general one on a narrow job while costing a thousandth as much to run.

Other task strings worth knowing: text-generation, summarization, translation, question-answering, zero-shot-classification, image-classification, automatic-speech-recognition.

When should you stop using pipeline() and drop a level?

When you need control over something it hides: how the model is compressed when loading, where it is placed, how requests are batched, seeing the raw scores, streaming the answer word by word, or using one loaded model for several jobs without loading it twice.

Notice the middle path this module actually takes: load the tokenizer and model yourself, with full control, then wrap them in a pipeline. You get the settings you need and the convenience you want.

Abstractions that hide five steps also hide five failure modes. Which ones does pipeline() hide?

Quiet cutting. Input longer than the model's limit gets chopped, usually with no clear warning. Your model then answers confidently about a document whose second half it never saw.

Hidden default model. pipeline("sentiment-analysis") with no model named quietly picks a small English one. Fine in a tutorial, wrong the moment your text is Hindi or financial.

Label names. One model says "POSITIVE", another says "LABEL_1", another says "positive". Code that checks for one breaks on the next.

Where it runs is chosen for you — which is how you end up on the CPU without noticing.

The general rule: shortcuts make the working path shorter and the broken path harder to understand. Use them, but know what you would have written by hand.

AutoTokenizer — the model's private language

Models do not read text. They read numbers. The tokenizer is the dictionary that converts between the two — and every model family has its own dictionary.

Inspect any tokenizer in four lines
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained("gpt2")
text = "AI is transforming industries and the future of work."

print(tok.tokenize(text))              # human-readable pieces
print(tok(text)["input_ids"])          # what the model actually sees
print(len(tok(text)["input_ids"]))     # your billing unit / context spend
The rule you must not break

The tokenizer must come from the same model as the weights. Always use AutoTokenizer.from_pretrained(model_id) with the exact same ID you gave the model. Mixing them does not crash. It quietly produces nonsense, which is much worse than crashing, because it just looks like a bad model.

Why tokenization is a business concern, not a detail

  • It's your billing unit. API pricing and context limits are in tokens, not words.
  • It is unfair across languages. Most tokenizers were built mainly from English text. The same sentence in Hindi, Tamil or Arabic often uses 2–3 times more tokens than English — so it costs more, and less of it fits in the same window. If you are building for Indian users, measure this before you size anything.
  • It explains silly-looking failures. Counting letters and spelling go wrong because the model never sees letters. Maths is shaky partly because numbers get split into odd chunks.
  • It shapes how you split documents. Chunk sizes are measured in tokens, so you need the tokenizer to plan them properly.

Why does mismatching a tokenizer produce fluent nonsense rather than an error?

Because the model gets a perfectly valid list of numbers. It has no way to know they came from a different dictionary — they are all inside its own range, so nothing looks wrong. It does what it always does: predicts a sensible continuation of the numbers it was given.

This is an important type of bug: both sides agree on the format and disagree on the meaning. Everyone agrees it is "a list of numbers". Nobody checks what the numbers stand for. The same trap hides in currency codes, units of measurement and status codes.

Your Hindi-language support bot costs 2.5× more per conversation than the English one, on the same model. Explain it to your finance team, then fix it.

The explanation: you pay per token, and the tokenizer was built from the text it was trained on. A dictionary learned mostly from English stores common English words as one token each, but breaks Devanagari into many small pieces. Same meaning, more tokens, higher bill — and less conversation fits before it gets cut off.

The fixes, in order: choose a model whose tokenizer was trained on many languages (several Qwen and Gemma versions handle Indian scripts far better); measure tokens per message on your real traffic instead of guessing; shorten your system prompt, since you pay for it on every single turn; and summarise old conversation instead of resending all of it.

The wider point if you build for Indian users: how efficiently a tokenizer handles your language is a real selection criterion — and it never shows up on any leaderboard.

AutoModelForCausalLM — loading the brain

Causal just means left to right. The model predicts the next word from everything before it, and can never look ahead. That is how every chat model works.

Sketch 9 · Causal generation is a loop, not a function
   prompt: "The capital of France is"
              |
              v
   +----------------------------+
   | forward pass through model |
   +----------------------------+
              |
              v   next-token probabilities
        Paris 0.91   the 0.03   a 0.02  ...
              |
              v   pick one (temperature / top_p decide how)
   "The capital of France is Paris"
              |
              +--------> feed the WHOLE thing back in
                         and predict the next token again
                         ... until max_new_tokens or a stop token

   Consequences that follow directly from this loop:
     * generation is inherently SEQUENTIAL -> can't parallelise a
       single response, which is why latency scales with output length
     * the model never revises. A wrong third token is now part of
       the "evidence" for token four. Errors compound forwards.
     * the KV cache exists so you don't recompute the whole prefix
       every step - that's the memory cost from section 06

Loading succeeded and the GPU is barely used, but generation takes 90 seconds per answer. Diagnose it.

device_map="auto" almost certainly pushed part of the model onto the CPU or the disk, because it did not fit in GPU memory. Any layer sitting in normal RAM has to be shuttled back and forth on every pass, which is hundreds of times slower.

Check with print(model.hf_device_map). If you see cpu or disk in there, make it fit: compress more, pick a smaller model, or get a bigger card. "It loaded" and "it is on the GPU" are two different statements, and only one of them is fast.

Your security team asks whether downloading models from Hugging Face is different from installing a Python package. Answer them.

It is the same kind of risk, but bigger and harder to inspect. A pip package is code you can read. A model is gigabytes of numbers you cannot read, plus sometimes code you can.

Same risks as packages: code that runs when you load the file, fake lookalike repositories, and versions changing under you because you did not pin them.

Risks packages do not have: the weights themselves can be poisoned — trained to behave normally except when they see a secret trigger phrase. No file scan will ever find that. You cannot read intent out of a pile of numbers.

So the controls are: safetensors only; pin exact versions; copy models into your own internal store after scanning; and test the behaviour, not just the file, for anything touching sensitive data. Be honest that this last check is imperfect — which is itself a good reason to stick to a few models from major labs rather than a long tail of community versions.

The reasoning behind temperature 0.2 here

In Section 5 you wanted variety — a poem, a landing page, a few different designs. Here you want the opposite: the answer that is actually in the document, the same way every time. A high temperature in a document Q&A system does not make it creative. It makes it wrong more often. And in finance, a made-up number written with confidence is the most dangerous thing your system can produce.

Match temperature to consequence: creative tasks tolerate variance, extraction tasks do not.

Same prompt, same model, different answer each run. Is that a bug?

Not by itself. That is do_sample=True doing exactly what you asked. Set do_sample=False and the same input gives almost the same answer every time.

Note "almost". Even then, small differences in GPU maths can change the result slightly. If you need the exact same text every time — for an audit trail or a test — the model alone cannot give you that. You save the answer and audit the saved copy. This is a real design limit that teams usually discover late.

Latency is proportional to output length, not input length. Why, and what does that imply for your UX?

The whole prompt is read in one go. But every word of the answer needs its own separate pass through the model. 500 words out means 500 passes, one after another, which cannot be done in parallel.

What that means for your app: stream the words as they appear, so the user feels the wait as "time until it starts" rather than "time until it finishes". Ask for short answers in the prompt — being brief is a speed setting, not just a style choice. And set max_new_tokens to what you will actually show; generating text nobody reads costs real seconds and real memory.

It also explains something surprising: feeding in a much longer document usually adds less time than asking for a slightly longer answer.

The failure that costs a week

If your PDF reader quietly mixes up table columns, your financial Q&A system will give you real-looking numbers attached to the wrong labels. No error, no warning, nothing. The model honestly reports what it was given — and what it was given was scrambled.

That is why the check above prints both the first 1,000 and the last 500 characters. Checking only the start is how you ship something that silently dropped the last eight pages.

When pypdf isn't enough

  • Scanned documents → you need OCR (text recognition from images). extract_text() just returns empty strings, and you must notice that rather than ignore it.
  • Tables that matter → use a proper table tool (camelot, pdfplumber) or a model that understands page layout. Never trust flattened table text for numbers.
  • Complicated layouts → use a reader that keeps the reading order and structure.
  • Any of the above at scale → build a real loading pipeline with quality checks per document, not a for-loop.
pypdf docs
Write three automated checks your ingestion step should run on every extracted document.

1. Is there enough text per page? Very few characters on a page almost always means it is a scan. Flag it for OCR instead of passing an empty string down the line.

2. Do the page counts match? The number of extracted pages should equal the number of pages in the file, and no page should be quietly empty in the middle.

3. Does the text look like text? Check that a sensible share of characters are letters. A wall of strange symbols means the font mapping failed.

And a fourth if numbers matter: check that something you know should be there — a heading, a date, a total — actually appears. Cheap, and it catches the scrambled-table case the other three miss.

Why is document ingestion, not the model, usually the biggest quality lever in a real RAG system?

Because everything after it is limited by it. The best model in the world reading scrambled text gives confident wrong answers. A modest model reading clean, well-split text gives good ones. No prompt can rescue broken input.

Teams get this backwards all the time — weeks comparing models, one afternoon on the PDF reader, then confusion about why accuracy stops improving. For most document AI projects the effort should go: clean input first, then retrieval, then the prompt, then the model. Model choice gets all the attention because it is the most fun to argue about.

      An instruction is a request, not a constraint
      
None of this is enforced. The model can still answer from what it already knows, especially about famous companies whose results it has memorised.       Your model answers a question correctly, but the answer is nowhere in the uploaded document. Bug or feature?
      

        
A bug — and a dangerous one, because being right hides it. The promise of your system is "answers come from this document". An answer from training data cannot be checked, cannot be cited, may be out of date, and will be confidently wrong the day your document disagrees with what the model memorised.

        
Worse, it makes your testing lie to you. You test on Google's earnings call, the model already knows Google's numbers, everything looks great. Then you point it at your own internal documents, which it has never seen, and accuracy collapses. Always include documents the model cannot possibly have memorised in your test set — ideally ones containing facts that go against common knowledge.

      

    


    

      Why does giving the model an explicit way to say "I don't know" improve accuracy more than telling it not to hallucinate?
      

        
Because the model's whole job is to produce a believable next word. "Don't make things up" tells it what not to do but gives it nothing to do instead. It still has to write something, and something believable is exactly what making things up looks like.

        
Giving it one specific sentence to use when the answer is missing gives it a real alternative to compete with invention. You have turned a vague ban into a simple choice between two paths.

        
This holds far beyond AI: saying what to do instead works better than saying what not to do. Same reason proper error handling beats telling a program "don't crash".

      

    


    

      You support four models behind a dropdown. Where does the chat-template logic belong in your codebase?
      

        
Nowhere in your code — that is the point. The template belongs to the model, so tokenizer.apply_chat_template() is the right dividing line. Adding a fifth model becomes a config entry, not a code change.

        
If you write the templates by hand, you have built a table that must be updated for every new model, goes stale the moment a vendor changes their format, and gives you worse answers instead of a clear error when it is wrong. That is the worst kind of failure: a config bug that looks like "this model seems dumber than the reviews said".

        
General rule: when a vendor ships instructions for how to talk to their thing, use their instructions instead of writing down your own copy.

      Why the tutorial is still right to truncate

Even without a vector database, you can do better

    
    
      
  • Split it up and pick by matching words. Cut the document into ~1,000-character pieces, rank them by how many words they share with the question, and send the best 5. No new tools, no database, and far more of the document becomes answerable.

  • Cut from the middle, not the end. Documents usually set the scene at the start and put conclusions at the end, so keeping both ends beats keeping only the start.
    •

      Why is truncating by characters rather than tokens a second, separate bug?
      

        
Because the model's limit is counted in tokens, and characters convert to tokens at different rates depending on the content and the language. 6,000 characters of English prose is about 1,500 tokens. The same 6,000 characters of tables, code or Hindi can be two or three times that. You have set your limit in the wrong unit.

        
The result: your safety margin is fake. It holds for the document you tested and quietly bursts through the limit on a different one — which then gets cut again, invisibly, inside the tokenizer. Always measure in the unit the system actually enforces.

      

    


    

      Your team wants document Q&A over 10,000 internal PDFs. Sketch the architecture, and name the parts this notebook doesn't have.
      

        
Loading: a pipeline that reads each PDF (falling back to OCR for scans), checks the quality, records where it came from (file, page, date, who may see it), splits it into overlapping pieces, and can be re-run when your reader improves.

        
Indexing: a model that turns text into vectors, a place to store them, and — important — keyword search alongside it, because vector search is poor at exact things like invoice numbers and ticker symbols.

        
Search: fetch the best few pieces, re-rank them, and filter by what this user is allowed to see, at search time. Most prototypes get this badly wrong. If search ignores permissions, your helpful Q&A bot happily summarises documents the user was never allowed to open.

        
Answering: the model, with page references it must include.

        
Testing: a list of questions with known answers, measuring two things separately — did search find the right piece, and did the answer stick to it. When quality drops you need to know which half broke.

        
Running it: re-indexing new documents, cost per question, a speed budget, and a log of who asked what and which documents came back.

      RAG adds retrieval failure modes on top of generation failure modes. Name two, and say how you'd detect them.
      

        
Search missed it. The right piece is in the document but was not found, so the model correctly says "not in the document". The output looks polite and well-behaved, and it is wrong. The unload sequence — order matters


import gc, torch

CURRENT = {"id": None, "pipe": None, "model": None, "tok": None}

def load_llm_model(display_name):
    model_id = AVAILABLE_MODELS[display_name]
    if model_id == CURRENT["id"]:
        return f"{display_name} already loaded."

    # --- unload: all three steps, in this order ---
    for k in ("pipe", "model", "tok"):
        CURRENT[k] = None          # 1. drop Python references
    gc.collect()                     # 2. actually collect them
    torch.cuda.empty_cache()         # 3. return the VRAM to the driver

    # --- load the new one ---
    tok = AutoTokenizer.from_pretrained(model_id)
    model = AutoModelForCausalLM.from_pretrained(
        model_id, quantization_config=bnb, device_map="auto")
    pipe = pipeline("text-generation", model=model, tokenizer=tok)

    CURRENT.update(id=model_id, pipe=pipe, model=model, tok=tok)
    return f"{display_name} loaded."

    


    

      Why del on its own does nothing
      
In Python, del only removes the name. The GPU memory is freed only when the last reference is gone, and the cleanup runs, and PyTorch hands the block back. Miss any one of the three and your "unloaded" model is still sitting in memory — which is exactly why the second model switch runs out of memory even though the first one worked. The three lines above are the fix.

    


    

      Also note: the dropdown label mapping is a real design choice
      
Mapping "Llama 3.2 (3B)" → "meta-llama/Llama-3.2-3B-Instruct" in a dictionary is not just for looks. It keeps the screen separate from the model list, so swapping the underlying model does not change what users see, and adding a model is one line of config. Small habit, but it is the difference between a demo and something you can maintain.

    


    

      Gradio docs
    


    

      Two users hit this Gradio app at once and one switches models. What happens?
      

        
Chaos. CURRENT is one shared variable with no protection. User A's question can get answered by user B's model, or by one that is only half loaded. Worse, a switch that happens mid-answer frees memory that a running request is still using.

        
This is not a Gradio problem. It is a design that assumes one user, now showing itself. The fix is not a lock bolted on afterwards. It is realising that loading models and handling requests belong in different layers. Real systems load models into long-running workers and send requests to them. They never load a model because someone clicked a dropdown.

"""Open-weight document Q&A. Colab: pip install -q transformers accelerate
bitsandbytes torch pypdf gradio"""
import gc, re, torch, gradio as gr
from pypdf import PdfReader
from transformers import (AutoTokenizer, AutoModelForCausalLM,
                          BitsAndBytesConfig, pipeline)

# ---------- 1. model registry: UI label -> hub id -----------------
MODELS = {
    "Phi-3 mini (3.8B)":  "microsoft/Phi-3-mini-4k-instruct",
    "Llama 3.2 (3B)":     "meta-llama/Llama-3.2-3B-Instruct",
    "Qwen 2.5 (3B)":      "Qwen/Qwen2.5-3B-Instruct",
    "Gemma 3 (4B)":       "google/gemma-3-4b-it",
}
BNB = BitsAndBytesConfig(load_in_4bit=True,
                         bnb_4bit_quant_type="nf4",
                         bnb_4bit_compute_dtype=torch.float16,
                         bnb_4bit_use_double_quant=True)
CUR = {"id": None, "pipe": None, "model": None, "tok": None}

# ---------- 2. PDF ingestion with a real sanity check -------------
def read_pdf(path):
    reader = PdfReader(path)
    pages = [(p.extract_text() or "") for p in reader.pages]
    empty = [i for i, t in enumerate(pages) if len(t.strip()) < 50]
    if empty:
        print(f"WARNING: pages {empty} look empty — scanned? needs OCR")
    return pages

# ---------- 3. retrieval-lite: rank chunks, don't truncate --------
def best_chunks(pages, question, size=1200, overlap=200, k=5):
    """Keyword overlap beats [:6000] and needs no vector DB."""
    chunks = []
    for pno, text in enumerate(pages, start=1):
        for i in range(0, len(text), size - overlap):
            body = text[i:i + size].strip()
            if body:
                chunks.append((pno, body))

    q = set(re.findall(r"\w+", question.lower())) - {
        "the", "a", "of", "is", "in", "what", "does", "and"}
    scored = sorted(chunks,
        key=lambda c: len(q & set(re.findall(r"\w+", c[1].lower()))),
        reverse=True)[:k]
    scored.sort(key=lambda c: c[0])                # restore doc order
    return "\n\n".join(f"[page {p}]\n{t}" for p, t in scored)

# ---------- 4. model lifecycle ------------------------------------
def load_model(label):
    mid = MODELS[label]
    if mid == CUR["id"]:
        return f"{label} already loaded."
    for key in ("pipe", "model", "tok"):
        CUR[key] = None
    gc.collect(); torch.cuda.empty_cache()

    tok = AutoTokenizer.from_pretrained(mid)
    mdl = AutoModelForCausalLM.from_pretrained(
        mid, quantization_config=BNB, device_map="auto")
    CUR.update(id=mid, tok=tok, model=mdl,
               pipe=pipeline("text-generation", model=mdl, tokenizer=tok))
    return f"{label} loaded."

# ---------- 5. the grounded answer --------------------------------
SYSTEM = ("You answer questions about a document. Use ONLY the document "
          "text below. If the answer is not there, reply exactly: "
          "'The document does not contain information on that topic.' "
          "Cite the page number you used. Do not use prior knowledge.")

def answer(pages, question):
    if CUR["pipe"] is None:
        return "Load a model first."
    context = best_chunks(pages, question)
    messages = [
        {"role": "system", "content": f"{SYSTEM}\n\nDOCUMENT:\n{context}"},
        {"role": "user",   "content": question},
    ]
    prompt = CUR["tok"].apply_chat_template(
        messages, tokenize=False, add_generation_prompt=True)
    out = CUR["pipe"](prompt, max_new_tokens=400, do_sample=True,
                       temperature=0.2, top_p=0.9,
                       return_full_text=False)          # no string slicing
    return out[0]["generated_text"].strip() or "(empty response)"

# ---------- 6. UI --------------------------------------------------
PAGES = read_pdf("earnings_call.pdf")

with gr.Blocks() as demo:
    gr.Markdown("# PDF Q&A — open-weight models")
    pick   = gr.Dropdown(list(MODELS), value=list(MODELS)[0],
                         label="Model")
    status = gr.Textbox(label="Status", interactive=False)
    q      = gr.Textbox(label="Your question")
    btn    = gr.Button("Ask", variant="primary")
    ans    = gr.Textbox(label="Answer", lines=8)

    pick.change(load_model, inputs=pick, outputs=status)
    btn.click(lambda question: answer(PAGES, question),
              inputs=q, outputs=ans)

demo.launch(debug=True)

    


    

      Four changes from the tutorial, and why
      
1. apply_chat_template instead of hand-written <|system|> tokens — so all four models work without any per-model formatting code.

      2. best_chunks instead of [:6000] — the whole document becomes answerable, with no new tools needed.

      3. return_full_text=False instead of searching for "Answer:" — a real boundary instead of a guess based on content.

      4. Page numbers in the text and an instruction to cite them — so answers can be checked.

  ================================================================
   THE MEMORY FORMULA  (memorise this one)
  ================================================================
   VRAM for weights (GB)  ~=  params(B)  x  bytes-per-weight
     fp32 = 4      fp16/bf16 = 2      int8 = 1      int4 = 0.5

     7B fp16 = 14 GB  -> will NOT fit a 15GB T4
     7B int4 = 3.5 GB -> fits easily

   ...then add KV cache, which grows with CONTEXT x BATCH.
   Weights are the floor. Your ceiling is set by your WORST request.

  ================================================================
   THREE LEVELS OF CONTROL
  ================================================================
   pipeline("task")           easiest. hides truncation, defaults,
                              label naming, device placement
   AutoTokenizer +            the working level: you control
   AutoModelForCausalLM       quantization + placement, then wrap
     -> then pipeline(...)    them in a pipeline
   raw generate()             when you need logits or streaming

  ================================================================
   THE THREE-STEP LOAD
  ================================================================
   model_id  = "org/Model-size-context-instruct"
   tokenizer = AutoTokenizer.from_pretrained(same id)
   model     = AutoModelForCausalLM.from_pretrained(
                   model_id, quantization_config=bnb,
                   device_map="auto")

   NEVER mix a tokenizer from one model with weights from another.
   It does not error. It produces fluent nonsense.

  ================================================================
   PROMPTING AN OPEN MODEL
  ================================================================
   messages = [{"role":"system",...},{"role":"user",...}]
   prompt = tokenizer.apply_chat_template(
              messages, tokenize=False, add_generation_prompt=True)

   Do NOT hand-write <|system|> / <|im_start|> / <start_of_turn>.
   Every family differs; the tokenizer already knows.

   Grounding recipe: "answer ONLY from the document; if it isn't
   there say <exact string>; cite the page; no prior knowledge."
   -> give the model a licensed way to FAIL, or it will invent.

  ================================================================
   DECODING SETTINGS
  ================================================================
   document Q&A ->  temperature 0.1-0.2, top_p 0.9, max_new 300-500
   creative     ->  temperature 0.7-1.0
   latency scales with OUTPUT length, not input length -> stream

  ================================================================
   THE BIG ARCHITECTURAL TRUTH
  ================================================================
   context = document[:6000]      <-- this is not a system.
                                      it is a placeholder for
                                      CHUNK -> EMBED -> RETRIEVE
                                      -> RERANK -> CITE

   In document AI, effort should go:
     ingestion quality > retrieval > prompt > model choice
   Most teams do it exactly backwards.

  ================================================================
   GOTCHAS
  ================================================================
   * del alone does NOT free VRAM. del + gc.collect()
     + torch.cuda.empty_cache() - all three, in that order
   * device_map="auto" can silently offload to CPU -> 50x slower.
     check model.hf_device_map
   * base vs instruct: base models do not follow instructions
   * trust_remote_code=True runs their Python on your machine
   * pypdf silently returns "" for scanned pages, and scrambles
     multi-column layouts and tables
   * truncating by CHARACTERS when the limit is in TOKENS
   * Colab idles out and wipes everything. That is not a bug,
     it is a reminder that a notebook is not infrastructure.

Glossary

    

      
Open weights
Downloadable model parameters you can run, inspect and fine-tune. Not the same as open source, which would also mean public training data and code.

      
Hugging Face Hub
Public registry of models, datasets and demo Spaces. The npm of machine learning.

      
Model card
The README of a model: licence, size, context, intended use, benchmarks, limitations. Read the licence before the benchmarks.

      
Gated model
Requires accepting terms and authenticating with a token before download. Llama models are gated.

      
Base vs Instruct
Base models continue text. Instruct/it/chat models have been further trained to follow requests. Use instruct for anything conversational.

      
Parameters (B)
Billions of learned weights. Predicts VRAM need, latency and roughly capability.

      
Quantization
Storing weights at lower precision (int8, int4) to shrink memory. Lossy, but neural networks tolerate it remarkably well.

      
NF4
A 4-bit format designed for the bell-shaped distribution of neural network weights. Better quality than naive int4 at the same size.

      
bitsandbytes
The library that performs on-the-fly quantization when loading a model in Transformers.

      
VRAM
GPU memory. The binding constraint in self-hosting — capacity stops you before speed does.

      
KV cache
Cached attention keys and values for the sequence so far. Grows with context length and batch size; the hidden memory cost.

      
Transformers
Hugging Face's Python library for loading and running models. Built on the 2017 "Attention Is All You Need" architecture.

      
Pipeline
High-level wrapper that chains tokenizer → model → post-processing into one call.

      
Tokenizer
Converts text to integer IDs and back. Model-specific — must always match the checkpoint.

      
Token
The unit a model reads. Roughly ¾ of an English word; considerably less efficient for Indic scripts.

      
Causal LM
Left-to-right next-token prediction. The architecture behind all generative chat models.

      
Chat template
The exact special-token format a model was fine-tuned on. Retrieve it with apply_chat_template; never hand-write it.

      
Special tokens
Markers like <|user|> or <|im_start|> that tell the model whose turn it is.

      
device_map="auto"
Lets accelerate place layers across GPU, CPU and disk. Convenient, and a silent performance trap when VRAM is short.

      
trust_remote_code
Executes custom Python shipped in the model repo. A security decision, not a convenience flag.

      
safetensors
A checkpoint format that cannot execute code on load, unlike legacy pickle files. Always prefer it.

      
Grounding
Constraining answers to a supplied source rather than the model's training data.

      
RAG
Retrieval-Augmented Generation: chunk, embed, retrieve the relevant pieces, then generate from them. What [:6000] is standing in for.

      
Chunking
Splitting a document into overlapping pieces sized to your context budget. Overlap prevents answers being severed at boundaries.

      
Continuous batching
Serving technique (vLLM, TGI) that packs requests together dynamically. The difference between 3 and 50 concurrent users on one GPU.

      
Gradio
Python library for building quick web UIs around models. Excellent for demos; not a production serving layer.

      
OCR
Optical Character Recognition — needed when a PDF is a scanned image and extract_text() returns nothing.
