Know the field

01

The big picture

That is it. Everything else — leaderboards, benchmarks, latency, cost — exists to help you answer one question before you even start: which model should I send this to?

Two things you learn at the same time

Skill A

Judging models

Reading leaderboards, understanding what a benchmark actually tests, knowing which numbers matter for your job and which are noise.

Skill B

Driving models

Writing a structured prompt, calling three different SDKs, cleaning the output, saving it, and improving it in plain English.

Why bother with three providers

Different models are good at different things. The one that writes the cleanest maths is often not the one that writes the nicest HTML. Also: one provider means one point of failure for outages, price hikes and rate limits. Wiring up a second one turns an emergency into a config change.

Your manager asks: "we already use one AI provider, why add another?" Give three answers.

1. Right tool per job. Route coding to the best coder, cheap bulk work to the cheap fast model, hard reasoning to the frontier model.

2. Cost. Similar quality models can differ many times over in price per million tokens. You can't use that unless you're connected to both.

3. Risk. Outages, rate limits, deprecations and price changes happen on the vendor's schedule, not yours. A tested second path means you migrate in a day, not a quarter.

02

Open models vs closed models

First fork in the road. This is not about which is smarter. It's about what you are allowed to hold in your hands.

Sketch 2 · Where the brain lives
  CLOSED  (GPT-4o, Claude, Gemini)     OPEN WEIGHTS  (Llama, Gemma, DeepSeek)

   your code                                 your code
      |                                          |
      | HTTPS request                            | local call
      v                                          v
  +---------------+                        +------------------+
  | VENDOR CLOUD  |                        | YOUR SERVER/GPU  |
  |  +---------+  |                        |  +------------+  |
  |  | weights |  |  <-- you can never     |  |  weights   |  | <-- you downloaded
  |  |  (secret)| |      see or copy       |  | (on disk)  |  |     these yourself
  |  +---------+  |                        |  +------------+  |
  +---------------+                        +------------------+
      |                                          |
   pay per token                            pay for hardware
   zero ops                                 you own all the ops
   data leaves your network                 data never leaves
Closed · you rent an endpoint

Easy, strong, not yours

You get frontier quality with zero infrastructure. Someone else handles scaling and upgrades. But you pay per token, you can't inspect the model, and your data goes to their servers.

Open weights · you own a copy

Free-ish, private, your problem

Download from Hugging Face, run it on your own machine, fine-tune it on your own data, keep everything inside your network. In exchange you own the GPUs, the serving, the scaling and the upgrades.

One word that matters

"Open weights" is not "open source." Most open models publish only the trained numbers, under a licence the vendor wrote. The training data and training code usually stay private, and some licences restrict commercial use. So you can run it and fine-tune it — you generally can't rebuild it. In a bank or any regulated shop, read the licence before you read the benchmark scores.

When is an open-weight model the only option, regardless of score?

When the data cannot legally leave your network. If customer PII, medical records or regulated financial data can't be sent to a third-party API, self-hosting is the only path — and any benchmark gap becomes a constraint you design around, not a reason to reject it.

Token maths you should do in your head

A token is a chunk of text. Roughly ¾ of an English word, or about 4 characters.

Sketch 3 · The token ruler
  text:     "Comparing"   "large"   "language"   "models"   "is"   "fun"
  tokens:   [Compar][ing]  [large]  [lang][uage]  [models]  [is]  [fun]
              1     2         3       4     5        6       7     8

  RULE OF THUMB
  ---------------------------------------------------------------
  1 token   ~= 0.75 word   ~= 4 characters
  1,000 tok ~= 750 words   ~= 1.5 pages
  1,000,000 tok ~= 750,000 words ~= 8-10 full novels in ONE prompt

  SPEED EXAMPLE
  2,500 tokens/sec  ->  ~1,875 words/sec  ->  a full essay every second

Context window vs max output — the confusion that costs you a day

Sketch 4 · Two different limits
  +====================== CONTEXT WINDOW (e.g. 1,000,000) ======================+
  |                                                                            |
  |  [ system prompt ][ your documents ][ chat history ][ your question ]       |
  |                                                     +--------------------+  |
  |                                                     |  MAX OUTPUT 8,000  |  |
  |                                                     |  the answer fits   |  |
  |                                                     |  in HERE only      |  |
  |                                                     +--------------------+  |
  +============================================================================+

  So a "1M context / 8K output" model is a brilliant reader and a short writer.
  Feed it a whole book, get back a few pages. Never more.

Latency vs speed — also two different things

Sketch 5 · What the user actually feels
  FAST-START MODEL (chat feel)
  press enter |--0.3s--| w o r d s   s t r e a m i n g . . . . done
              ^ latency ^ <----------- throughput ----------->

  REASONING MODEL (thinks first)
  press enter |----------- 30s thinking -----------| words words done
              ^ ------- latency (TTFT) ----------- ^

  For a CHAT UI      -> latency wins. Users abandon a frozen screen.
  For a BATCH JOB    -> latency is irrelevant. Only throughput + cost matter.
  Slow first token is often a feature: the model is reasoning, not stuck.
A model advertises a 1,000,000-token context. Name two things that number does NOT tell you.

1. How much it can write back. Max output is a separate, far smaller limit.

2. How well it uses the middle. A big window is a hard cap, not a promise of perfect recall — models often miss detail buried in the middle of a giant prompt. And every extra token you stuff in costs money and adds latency.

04

Benchmarks in plain words

Leaderboard columns are acronyms. Each one is a specific exam with a specific blind spot. Know what the exam tests, or you're just being sold a number.

Reasoning

GPQA Diamond

"Google-proof" science questions in biology, physics and chemistry, at PhD level. Searching the web doesn't save you. Course snapshot: leaders around 84–85% (Grok ≈ 84.6, Gemini 2.5 Pro ≈ 84).

Maths

AIME / MATH-500

Competition-level maths. Not arithmetic — multi-step problems. Leaders reached the low 90s (o4-mini ≈ 93.4%). Remember these same models were terrible at maths in 2022.

Coding

SWE-bench

Real GitHub bugs. The model must write a patch that makes the test suite pass. Closest public proxy for agentic coding, not snippet writing. Claude 3.7 Sonnet led at ≈ 70.3%.

The ceiling

Humanity's Last Exam

~2,500 brutally hard expert questions, designed so nobody aces them soon. Scores were still low (o3 ≈ 20.3%), which is exactly why it's useful — it still has room to grow.

Sketch 6 · Which exam predicts which job
  YOUR TASK                        LOOK AT THIS COLUMN
  --------------------------------------------------------------
  "fix the failing test in repo"   ->  SWE-bench
  "check this actuarial formula"   ->  MATH-500 / AIME
  "explain this research paper"    ->  GPQA Diamond
  "will it survive weird edge Qs"  ->  Humanity's Last Exam
  "will users enjoy the answer"    ->  none of these -> LMArena (sec 06)
  "will it work on MY data"        ->  none of these -> your own eval set
Four ways a benchmark number lies to you

Contamination — the test questions may have leaked into training data.
Saturation — once everyone scores 92–94%, you're measuring noise, not skill.
Scaffold sensitivity — agent scores depend heavily on the wrapper around the model, so two numbers may not measure the same system.
Self-reporting — vendor numbers use vendor-chosen settings.

Use leaderboards to build a shortlist of 3. Never to declare a winner.

Two models are 1.2 points apart on a benchmark where the leader scores 93%. What do you conclude?

Almost nothing. That's saturation — the gap is inside the noise created by prompt formatting, temperature, harness differences and leftover contamination. Decide on the axes that still separate them: cost, latency, context, output ceiling, tool reliability, and your own test set.

05

Vellum Leaderboard — the spec-sheet view

Vellum collects published benchmark scores for models released after April 2024, plus the operational numbers most benchmark sites skip: latency, speed, price, context, cutoff.

How to actually use it

  1. Top cards — best in reasoning (GPQA), best in high-school maths, best in agentic coding (SWE-bench). Hover any card to read what the benchmark means instead of guessing from the acronym.
  2. Independent evals — extra metrics including Humanity's Last Exam. Useful because the headline benchmarks are saturated.
  3. Speed & cost tabs — fastest by tokens/second, lowest latency by seconds-to-first-token, cheapest by input and output price. This is half the decision and most leaderboards ignore it.
  4. Two-model compare — pick any two, read context, cutoff, I/O cost, max output, latency and speed side by side. (That's what the bench in section 07 recreates.)
  5. The big sortable table — the real tool. Sort by MATH-500 for the best mathematician, SWE-bench for the best coder, GPQA for the best scientist. Different sort, different winner, every time.
Expect blank cells

Not every vendor publishes every score — in the course walkthrough, Gemini's maths row was simply empty. A blank is not a zero. It means "not reported." Never rank on a column full of holes without saying so out loud.

What does Vellum give you that a pure capability leaderboard doesn't?

The operational columns — latency, throughput, input/output price, context window, max output and cutoff — sitting next to the benchmark scores. Those decide whether a model is deployable at your volume and your response-time budget, which is a completely separate question from whether it's smart.

Where human voting misleads

Style bias: longer, prettier, more confident answers win votes even when they're less correct.
Prompt mix: casual chat prompts dominate the vote pool, so the ranking reflects chat quality, not domain work.
Fact-checking: a voter can't catch an error about something they don't know.

Read Arena as "which model do people enjoy reading" and Vellum as "which model gets it right." You need both.

Vellum ranks A above B. The Arena ranks B above A. Who's wrong?

Neither — they measure different things. Vellum = benchmarked correctness on structured exams. Arena = human preference on open-ended everyday prompts, where tone and formatting weigh heavily. A model can be more accurate and less pleasant, or the reverse.

The disagreement is information: it tells you the two models fail in different places, and that you should build your own domain test set to break the tie.

What this pairing teaches

You must summarise 400-page regulatory PDFs overnight, in bulk. Which axes decide it? Which are noise?

Decisive: context window (does the doc fit, or must you chunk it?), max output (how long can the summary be?), input price per 1M tokens (you're pushing enormous prompts, so input cost dominates the bill), throughput.

Noise: time-to-first-token. It's an overnight batch — nobody is watching the cursor. Paying extra for responsiveness you'll never perceive is pure waste.

08

Keys, .env, and not getting a shock bill

Three providers, three consoles, three keys. Each one is generated once in a developer console and usually shown only once. Copy it straight into your .env file.

.env — never commit this
# Add .env to .gitignore BEFORE you create it.
OPENAI_API_KEY=sk-...
GOOGLE_API_KEY=AIza...
ANTHROPIC_API_KEY=sk-ant-...
Rule 1

Never in the notebook

Keys live in .env or a vault. Not in a cell, not in a commit, not in a screenshot of your terminal. A leaked key is someone else's compute on your invoice.

Rule 2

Cap the damage

Set hard spend limits and usage alerts in every console on day one. Use a separate key per project so one can be killed without breaking the rest.

Rule 3

Check, don't print

To confirm a key loaded, print the first 5 characters and the length. That proves it resolved without writing the secret into your output cells and logs.

Load and sanity-check all three
import os
from dotenv import load_dotenv

load_dotenv(override=True)

keys = {
    "OpenAI":    os.getenv("OPENAI_API_KEY"),
    "Google":    os.getenv("GOOGLE_API_KEY"),
    "Anthropic": os.getenv("ANTHROPIC_API_KEY"),
}

for name, k in keys.items():
    if not k:
        print(f"{name}: MISSING")              # fail loudly, early
    else:
        print(f"{name}: {k[:5]}… ({len(k)} chars)")
The expensive mistake

A key pushed to a public repo is typically scraped and used within minutes — bots watch GitHub's event stream. If it happens: revoke first, investigate second. Rotating a key takes thirty seconds. Disputing a bill does not.

Why print only the first five characters of a key?

Because the question is "did the environment variable load?" — not "what is my secret?" Prefix plus length proves it worked. Printing the whole thing writes the secret into notebook outputs, terminal scrollback and log aggregators, all of which outlive your session and get shared.

Wrap it once, use it forever

After writing those three calls, the obvious next step is one function that hides the difference. This is what makes routing, A/B testing and failover possible later.

A minimal router
def ask(provider, prompt, **kw):
    """One signature, three back-ends. Always returns plain text."""
    if provider == "openai":
        r = oai.chat.completions.create(
            model=kw.get("model", "gpt-4o"),
            temperature=kw.get("temperature", 0.5),
            messages=[{"role": "user", "content": prompt}])
        return r.choices[0].message.content

    if provider == "gemini":
        r = gem.models.generate_content(
            model=kw.get("model", "gemini-2.5-pro"), contents=prompt)
        return r.text

    if provider == "claude":
        r = ant.messages.create(
            model=kw.get("model", "claude-sonnet-4-5"),
            max_tokens=kw.get("max_tokens", 20000),
            messages=[{"role": "user", "content": prompt}])
        return r.content[0].text

    raise ValueError(f"unknown provider: {provider}")
You port a working prompt from OpenAI to Anthropic and it errors before reaching the model. First thing to check?

max_tokens. Anthropic requires it — there's no default. It's the single most common porting error. Second most common: reaching for .choices[0].message.content on a reply whose text actually lives at .content[0].text.

10

Three probe prompts to test any model in five minutes

Three deliberately different prompts. Each catches a different failure. Run them on any new model before trusting it.

Probe 1 — maths (one right answer)

"A father is 36 and his son is 6. In how many years will the father be exactly five times as old as his son?"

The answer, so you can mark it instantly
  36 + n = 5 x (6 + n)
  36 + n = 30 + 5n
       6 = 4n
       n = 1.5 years

  check:  father 37.5,  son 7.5,  37.5 / 7.5 = 5  ✓
Why the answer being 1.5 is the whole point

It's not a whole number. Models that pattern-match toward "clean" answers get caught. Models that silently round get caught. A probe whose correct answer looks slightly wrong is worth ten probes whose answer looks right.

Probe 2 — creativity (no right answer)

"Write a funny poem for my niece Rose, who is turning one." You're judging tone control, whether all three constraints (name, age, humour) are honoured, and whether it reaches for cliché. This is exactly where human preference beats any benchmark.

Probe 3 — code

"Write a Python function that calculates BMI, letting the user pick metric or imperial units." Look past whether it runs:

  • Does it handle the unit switch cleanly, or bolt it on?
  • Does it raise an error on an unknown unit, or return something wrong and silent?
  • Docstring and edge cases — or a wall of scaffolding you didn't ask for? More code is not better code.

A rubric you can actually score

CriterionAsk yourselfTested by
CorrectnessIs it right — and right again on a retry?Probe 1 & 3
Legible reasoningCan you follow and audit the working?Probe 1
Instruction fidelityWere all stated constraints honoured?All three
ConcisionIs the length proportional to the task?Probe 2 & 3
Format disciplineClean, or noisy stray markdown?All three
Felt latencyHow long until the first word?All three
Cost per runTokens in + out at that model's rateAll three
The habit that actually pays

Three generic probes tell you nothing about your work. Build a fixed set of 20–30 prompts from real tasks in your domain, with known-good answers where possible, and run every candidate model against it. That private set is the only leaderboard about your problem — and it stays useful as models change, which no public score does.

Model A answers correctly in one dense paragraph. Model B answers correctly with the equations laid out. Same cost, same speed. Which do you ship?

Depends who reads it — and that is the answer. If a human must audit the working (a compliance calculation, a customer explanation), B's legibility is the product. If code parses the output and only the final number matters, A is cheaper and easier to extract from, and B's working is noise you'd strip.

General lesson: "which model is better" has no answer. "Better for this reader, this task, this budget" always does.

Part three

Build the landing page

Now the fun part. You describe a website in plain English, three models write the HTML, and you compare. This is the section people call "vibe coding" — but it works far better when the prompt has a real structure.

11

Anatomy of the prompt

The same prompt goes to all three models, so it has to be complete on its own. A good prompt has three parts.

Sketch 11 · The three-part prompt skeleton
  +--------------------------------------------------------------+
  | 1. CONTEXT   — who should the AI pretend to be?              |
  |    "You are a helpful AI assistant acting as a               |
  |     front-end web developer."                                |
  +--------------------------------------------------------------+
  | 2. INSTRUCTION — what exactly do you want done?             |
  |    "Generate the complete HTML for a clean, professional     |
  |     landing page for {startup_name}. Concept: {concept}."    |
  |    + the list of sections you want                           |
  +--------------------------------------------------------------+
  | 3. OUTPUT INDICATOR — what shape must the answer be?        |
  |    "Return ONLY raw HTML, starting with <!DOCTYPE html>      |
  |     and ending with </html>. Inline CSS. No images.          |
  |     No explanation, no markdown fences."                     |
  +--------------------------------------------------------------+

  Miss part 3 and you get: "Sure! Here's a landing page for you:"
  followed by ```html ... ``` — which breaks your saved file.

Why each rule in the prompt exists

InstructionReason it's there
"acting as a front-end web developer"Sets the role. Pushes the model toward developer conventions instead of generic prose.
"start with <!DOCTYPE html>, end with </html>"Gives an exact, checkable shape for the output. Easy to validate in code.
"don't include images"Models invent image URLs that don't exist, so you get broken icons everywhere. Removing images removes the problem.
"raw HTML with inline CSS"Everything lands in one file. No separate stylesheet to wire up. Open it and it just works.
Explicit section listWithout it, each model invents a different page structure and you can't compare them fairly.
"no explanation"Stops the model wrapping the code in chatty commentary you'd have to strip out.

  |   Name | Email | Message   [ SEND ]  (red button)        |  <- contact form
  +----------------------------------------------------------+
  |   links · links · social icons · copyright               |  <- footer
  +----------------------------------------------------------+
Real-world caution about those testimonials

Asking for fictional customer quotes is fine for a learning exercise. Never ship AI-invented testimonials on a live product page. Fake endorsements are treated as deceptive advertising in most jurisdictions, including under India's consumer-protection and ASCI rules. Replace them with real quotes or a placeholder before anything goes public.

Why does the prompt say "don't include images"?

Because the model can't upload files — it can only write image tags pointing at URLs it invents. Those URLs almost never exist, so you get a page full of broken-image icons. Banning images gives you a page that looks finished on first open. (Later you can allow inline SVG or emoji, which need no external file — that's part of why some models' pages looked richer.)

The generation pipeline

Getting text back is only step one. There are three more steps before you have a page you can open, and the middle one is where everybody trips.

  |  send the |  text    | strip the  | html  | write to |    | open in |

  What the model often returns:        What you need on disk:

  <!DOCTYPE html>              ---->     ...

The clean + save helper (write once, use for all three)
import re, pathlib
from IPython.display import display, HTML, Markdown

def clean_html(raw: str) -> str:
    """Remove markdown fences and any chat before/after the HTML."""
    txt = raw.strip()
    txt = re.sub(r"^```(?:html)?\s*", "", txt)   # leading  ```html
    txt = re.sub(r"\s*```$", "", txt)            # trailing ```
    start = txt.find("<!DOCTYPE")
    if start == -1:
        start = txt.find("<html")
    end = txt.rfind("</html>")
    if start != -1 and end != -1:
        txt = txt[start:end + 7]                # keep only the page
    return txt.strip()


def save_and_show(raw: str, filename: str, preview=True):
    html = clean_html(raw)
    path = pathlib.Path(filename)
    path.write_text(html, encoding="utf-8")
    print(f"saved -> {path.resolve()}  ({len(html):,} chars)")
    if preview:
        display(HTML(html))
    return html

Now run all three, with timing and cost visible

The comparison loop — this is the whole project
import time

runs = [
    ("openai", "gpt-4o",           "openai_landing_page.html"),
    ("gemini", "gemini-2.5-pro",   "gemini_landing_page.html"),
    ("claude", "claude-sonnet-4-5","claude_landing_page.html"),
]

results = []
for provider, model, filename in runs:
    print(f"\n=== calling {provider} ({model}) ===")
    t0 = time.time()
    try:
        raw = ask(provider, html_prompt, model=model)
        elapsed = time.time() - t0
        html = save_and_show(raw, filename, preview=False)
        results.append({"provider": provider, "model": model,
                        "seconds": round(elapsed, 1),
                        "chars": len(html), "file": filename})
    except Exception as e:
        print(f"FAILED: {e}")        # one provider down != project down

import pandas as pd
display(pd.DataFrame(results))     # your own mini leaderboard
Small upgrade, big payoff

The course runs each provider in a separate cell. Putting them in one loop that records seconds and output size turns a demo into a measurement. After three or four startups you'll have real data on which model is fastest and most verbose for your prompts — which is worth more than any public leaderboard.

Why does the pipeline need a "clean" step at all?

Because models are trained to be chatty and to format code in markdown. Even when you say "only HTML," you'll often get ```html fences and a friendly sentence around the code. Written straight to a .html file, those characters appear on the page and can break parsing. The clean step slices out everything from <!DOCTYPE to </html> and throws the rest away.

The lesson hiding in the mini result

GPT-4o-mini beating GPT-4o on this task is the most instructive moment in the module. The bigger, more expensive model is not automatically better for your specific task. How to judge them properly, not just by eye

The final practice asks which model produced the cleanest, most semantic HTML — judged by reading the code, not the rendered page. That's a different and better question. Things to check:

  • Semantic tags — does it use <header> <nav> <section> <footer>, or is everything a <div>?
  • Heading order — one <h1>, then <h2>s in order? Or headings picked by how big they look?
  • Accessibility — alt text, labels on form inputs, sensible colour contrast.
  • Responsiveness — is there a viewport meta tag and any media query, or does it break on a phone?
  • CSS sanity — repeated inline blocks copy-pasted everywhere, or a consistent set of styles?
  • Dead weight — unused classes, empty tags, commented-out fragments.

Why is "which page looks best" a weaker question than "which HTML is cleanest"?

Because looks are the easy half. A page can be beautiful and still be unmaintainable div soup with no semantic tags, no accessibility and no responsiveness — which means a developer has to rewrite it before it can be used. Clean semantic HTML can be restyled in an afternoon. Judge the artefact you'd have to maintain, not the screenshot.

Vague vs specific — the same request, two results

Vague

"add a contact form with red colours to grab attention"

Works, and models understand it. But "red" is a guess — you'll get whatever red the model likes, different in each of the three pages, and hard to reproduce next run.

Specific

"add a contact section with name, work email and message fields. Submit button: background #E63946, white text, 8px radius."

Same effort to type. Now all three pages are comparable, the result is reproducible, and you're testing the model rather than testing your own vagueness.

Prompt upgrades worth adding to every run

Add this lineWhat it fixes
"Use semantic HTML5 tags: header, nav, main, section, footer."Stops div soup. Makes the output actually maintainable.
"Must be responsive down to 375px. Include a viewport meta tag."Otherwise the page breaks on a phone.
"Palette: #0B1F3A background, #F4A261 accent, #F8F9FA text."Removes colour guesswork; all three pages become comparable.
"All form inputs must have associated label elements."Basic accessibility, usually skipped unless asked.
"Use inline SVG or emoji for icons. No external image files."Icons appear without any broken links.
"Mark testimonials with a data-placeholder attribute."Makes fake quotes easy to find and replace before going live.
Change one thing at a time

If you add a contact form and change the colours and switch the model in the same run, you can't tell which change caused the difference. Same discipline as any experiment: one variable per run, and keep the old file (use distinct filenames — see the next section).

Why give a hex code instead of just saying "red"?

Because "red" is a free variable each model fills in differently, and differently again on the next run. With a hex code, the colour stops being a source of variation, so any difference you see between the three pages is caused by the model, not by your ambiguity. Same principle as fixing the temperature or the seed: control everything you're not testing.

Gotchas and fixes

Everything that goes wrong in this project, and what to do about it.

Two habits worth keeping

Pin your model IDs in a config dict at the top of the notebook. When a model is retired you change one line, not fifteen.

Keep every generated file. Storage is free; a folder of dated outputs is the evidence base for "which model is actually better," and lets you re-open old runs after a model has changed underneath you.

compare_llms.py — end to end
"""Send one prompt to three providers, save three HTML pages, compare."""
import os, re, time, pathlib
from datetime import date
from dotenv import load_dotenv
from openai import OpenAI
from google import genai
import anthropic

load_dotenv(override=True)
oai = OpenAI()
gem = genai.Client()
ant = anthropic.Anthropic()

# --- 1. pin your models in ONE place -----------------------------
MODELS = {
    "openai": "gpt-4o",
    "gemini": "gemini-2.5-pro",
    "claude": "claude-sonnet-4-5",
}

# --- 2. the idea -------------------------------------------------
startup_name = "ConnectGenius"
startup_concept = (
    "An intelligent CRM that uses AI to analyse customer interactions, "
    "predict needs and automate personalised follow-ups."
)

# --- 3. the prompt template --------------------------------------
html_prompt = f"""You are a helpful AI assistant acting as a front-end web developer.

Generate the complete HTML for a clean, professional landing page for
{startup_name}.

Concept: {startup_concept}

Sections: header with name and tagline; hero with headline and CTA;
3-4 features; how it works; testimonials (fictional, marked with
data-placeholder); pricing; contact form with labelled fields; footer.

Rules:
- Return ONLY raw HTML from <!DOCTYPE html> to </html>. No fences, no prose.
- Inline CSS only. Semantic HTML5 tags. Responsive to 375px.
- No external images; inline SVG or emoji for icons.
- Palette: #0B1F3A background, #F4A261 accent, #F8F9FA text.
"""

# --- 4. one call shape for all three ------------------------------
def ask(provider, prompt):
    if provider == "openai":
        r = oai.chat.completions.create(
            model=MODELS[provider], temperature=0.3,
            messages=[{"role": "user", "content": prompt}])
        return r.choices[0].message.content
    if provider == "gemini":
        r = gem.models.generate_content(model=MODELS[provider], contents=prompt)
        return r.text
    if provider == "claude":
        r = ant.messages.create(
            model=MODELS[provider], max_tokens=20000,
            messages=[{"role": "user", "content": prompt}])
        return r.content[0].text
    raise ValueError(provider)

# --- 5. clean the output -----------------------------------------
def clean_html(raw):
    t = re.sub(r"^```(?:html)?\s*|\s*```$", "", raw.strip())
    s, e = t.find("<!DOCTYPE"), t.rfind("</html>")
    return t[s:e + 7].strip() if s != -1 and e != -1 else t.strip()

# --- 6. run, time, save, report ----------------------------------
out = pathlib.Path("pages"); out.mkdir(exist_ok=True)
stamp = date.today().isoformat()
rows = []

for provider in MODELS:
    t0 = time.time()
    try:
        html = clean_html(ask(provider, html_prompt))
        f = out / f"{provider}_{stamp}.html"
        f.write_text(html, encoding="utf-8")
        rows.append((provider, MODELS[provider],
                     round(time.time() - t0, 1), len(html), str(f)))
    except Exception as e:
        rows.append((provider, MODELS[provider], None, None, f"FAILED: {e}"))

print(f"{'provider':10}{'seconds':>9}{'chars':>9}  file")
for p, m, secs, n, f in rows:
    print(f"{p:10}{str(secs):>9}{str(n):>9}  {f}")

Glossary

Token
Chunk of text a model reads and writes. ≈ ¾ of a word, ≈ 4 characters. Billing, limits and speed are all counted in these.
Context window
Total tokens allowed in one call — prompt and answer together.
Max output tokens
Separate cap on how long one answer can be. Usually far smaller than the context.
TTFT / latency
Seconds until the first token appears. What users feel as responsiveness.
Throughput
Tokens per second once it starts writing. Sets total completion time.
Knowledge cutoff
Last date in the training data. Anything newer must come from search or your documents.
Parameters
Learned weights in the network. More capacity — but also more cost and latency.
Temperature
Randomness dial. Low = repeatable, good for code and comparisons. High = varied, good for creative work.
Frontier model
Informal name for the current top tier of models across providers.
Open weights
Downloadable parameters you can self-host and fine-tune. Not the same as open source.
GPQA Diamond
Graduate-level "Google-proof" science questions. The reasoning benchmark.
SWE-bench
Patch real GitHub issues so tests pass. The agentic-coding benchmark.
Humanity's Last Exam
~2,500 expert questions built to resist saturation. Still low scores, still useful.
Arena Score
Elo-style rating from blind head-to-head human votes on LMArena.
Saturation
When top scores cluster at the ceiling and remaining gaps are just noise.
Contamination
Test questions leaking into training data, inflating scores without real skill.
Style bias
Human raters preferring longer, prettier answers regardless of correctness.
Pareto frontier
On a cost-vs-quality chart, the models you can't improve on one axis without losing the other. The only sensible choices.
Markdown fences
The ``` markers models wrap code in. Must be stripped before saving to a .html file.
Inline CSS
Styles written directly in the HTML instead of a separate file — so the whole page is one self-contained file.
Semantic HTML
Using tags that describe meaning (header, nav, section, footer) rather than generic divs. Better for accessibility, SEO and maintenance.
Hero section
The big top area of a landing page: headline, tagline, main call-to-action button.
